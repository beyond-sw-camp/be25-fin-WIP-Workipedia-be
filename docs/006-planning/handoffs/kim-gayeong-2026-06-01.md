## 3. 서비스 흐름 변경 요약

기존에는 사용자가 질문을 하면 챗봇이 답변하고, 답변하지 못한 경우에 티켓으로 이어지는 흐름이 중심이었다.
현재 기획은 입력을 처음부터 `질문`과 `요청`으로 나눈다.

| 입력 | 처리 방식 | 관리자/포인트/ESG 영향 |
|---|---|---|
| 질문 | 챗봇이 매뉴얼/워키를 검색해 출처 기반 답변 | 챗봇 해결 건수, 출처 기반 답변 비율, 지식 재사용 지표 |
| 질문 후 미해결 | 챗봇 답변 없음/불충분/공식 처리 필요 시 요청 티켓으로 전환 | 티켓 전환 건수, 반복 문의 감소 지표 |
| 요청 | 챗봇을 거치지 않고 티켓으로 바로 발행 | 팀 큐/공통 접수 큐, 자동 배정 성공률, 처리 완료율 |
| 처리 완료 요청 | 담당자가 지식화 후보 등록, TEAM_ADMIN 검수 후 워키 반영 | 지식화 전환율, ESG 지식 공유 지표, 포인트/뱃지 이벤트 |

핵심:

```text
질문 = 기존 지식을 찾는 흐름
요청 = 실제 업무 처리가 필요한 티켓 흐름
```

김가영 담당 관리자 화면은 `요청 티켓`과 `지식화 후보`를 중심으로 운영 지표를 보여줘야 한다.
