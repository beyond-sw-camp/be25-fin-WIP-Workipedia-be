## 4. DB migration 논의 안건

최신 기획 기준으로 `feat/common-response` 브랜치에 DB migration 보강사항을 추가했다.
월요일 오전에는 아래 항목을 기준으로 각 도메인 개발자가 본인 엔티티 구현 전에 충돌 여부를 확인한다.

| 구분 | 추가/변경 내용 | 논의할 것 | 관련 담당자 |
|---|---|---|---|
| 카테고리 | `categories`, `department_category_mappings` 추가 | 요청 티켓 자동 배정에 사용할 초기 카테고리 seed와 부서 매핑 기준을 누가 관리할지 결정 | 김진혁, 김가영 |
| 티켓 상태 이력 | `ticket_status_logs` 추가 | 티켓 상태 변경 감사 로그를 이 테이블로 충분히 볼 수 있는지 확인 | 김진혁, 김가영 |
| 누적 포인트 | `user_points` 추가 | `point_history`는 이력, `user_points`는 현재값으로 분리하는 기준 확정 | 김가영 |
| 매뉴얼 상태 | `manuals.status` 추가 | `DRAFT`, `PUBLISHED`, `ARCHIVED`, `DELETED` 중 MVP에서 실제로 사용할 상태 확정 | 김진혁, 민정기 |
| 챗봇 출처 연결 | `worki_questions.source_chatbot_message_id` FK 추가 | 챗봇 실패 후 워키/요청 전환 시 원본 메시지를 반드시 남길지 확인 | 김진혁, 이슬이 |
| RAG 청크 출처 | `worki_chunks.source_type`, `source_id` 추가 | 질문 청크와 답변 청크를 같은 테이블에서 관리하는 방식 확정 | 김진혁, 민정기 |

### 월요일 결정해야 할 질문

1. 자동 배정용 카테고리의 1차 목록을 누가 만들고 어디에 seed로 둘 것인가?
2. `manual_versions`, `manual_citations`, `worki_search_logs`, `points_daily_limit`, `esg_grade`는 MVP DB에 포함하고, `esg_metric_snapshots`는 날짜별 추이 화면 전까지 제외한다.
3. 티켓 상태값은 `RECEIVED`, `COMMON_QUEUE`, `ASSIGNED`, `IN_PROGRESS`, `COMPLETED`를 기본으로 두고, `REJECTED`, `DELETED`는 관리용 상태로 허용할 것인가?
