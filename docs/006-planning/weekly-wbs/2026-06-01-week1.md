## 1.1 개발 시작 전 Issue

Week 1 개발은 Weekly Issue 하나로 묶지 않고 기능 Issue 단위로 만든다.
Weekly WBS는 일정 기준이며, 실제 GitHub Issue는 아래처럼 기능 단위로 생성한다.

```text
[feat] Auth 회원가입/로그인/JWT 1차 구현
[feat] 챗봇 세션/메시지 저장 API 구현
[feat] 티켓 생성/조회/상태/배정 API 구현
[feat] 워키 질문/답변/채택 API 구현
[feat] 관리자 큐/포인트/ESG 등급 기본 API 구현
[feat] 프론트 라우팅/주요 화면/API 연동 준비
```

Feature Issue에는 아래 형식을 사용한다.

```md
## 기능 설명
-

## 관련 도메인
-

## 권한
-

## 고려사항
-

## 예외 케이스
-

## 완료 기준
- [ ] 정상 동작 확인
- [ ] 데이터 정합성 확인
- [ ] 예외 케이스 처리 완료
- [ ] PR 연결 및 머지
```

## 2.1 황희수 FE 작업 기준

황희수는 FE skeleton 구현 전에 Figma Make 화면이 최신 기획 문서와 맞는지 먼저 확인한다.

확인 기준:

- 메인 진입이 `질문`과 `요청`으로 분리되어 있는가
- 질문 -> 챗봇 -> 매뉴얼 + 워키 출처 답변 흐름이 보이는가
- 답변 실패/불충분 시 요청 티켓 발행 액션이 있는가
- 요청 -> 티켓 발행 -> 자동 배정/공통 접수 큐 흐름이 표현되는가
- `TEAM_ADMIN`이 자기 팀 티켓을 확인하고 팀원에게 배정하는 화면이 있는가
- `SYSTEM_ADMIN`이 공통 접수 큐를 확인하는 화면이 있는가
- 처리 완료 티켓의 지식화 후보/검수 화면이 있는가
- 포인트/ESG 등급/지표 카드는 MVP 수준으로라도 노출되는가

수정 기준:

- 큰 화면 구조나 시연 동선 변경은 Figma Make에서 먼저 맞춘다.
- 버튼명, 문구, 상태 표시, API 연동에 따른 세부 UI는 FE 코드에서 맞춘다.
- Figma와 문서가 충돌하면 `docs/001-reference/service-flow.md`와 `docs/001-reference/project-proposal.md`를 기준으로 한다.

```text

```

예시:

```text
docs/006-planning/daily-reports/2026-06-02/kim-jinhyeok.md
```

작성 기준:

- 오늘 완료한 것
- 오늘 못 끝낸 것
- 내일 오전에 결정해야 하는 것
- API/DB/화면 영향이 있는 변경
