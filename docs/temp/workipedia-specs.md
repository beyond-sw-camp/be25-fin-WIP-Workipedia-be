볼륨 확대에 대한 고민 중, 기존 요청(티켓) / 질문(워키) 2채널에서 3채널 구조로 확장.

| 채널 | 특징 | 기록 |
|---|---|---|
| 요청 (티켓) | 가장 정확, 공식 처리 | 영구 보존 |
| 질문 (워키) | 중간, 커뮤니티 Q&A | 영구 보존 |
| **Flash Chat** | 가장 빠름, 가벼운 질문 | **10분 후 삭제** |

## 1. Flash Chat

### 개요
전사 단일 채널. 누구나 들어와서 가볍게 묻고 답하는 실시간 공개 채팅방.
10분 후 메시지 자동 삭제 (Redis TTL).

### 기술 스택
- **백엔드**: Spring WebSocket + STOMP (`spring-boot-starter-websocket`)
- **메시지 저장**: Redis (TTL 600초, DB 저장 없음)
- **브로드캐스트**: STOMP topic `/topic/flash-chat`

### 주요 기능
- 전사 공개 채널 (별도 입장 절차 없음)
- 원클릭 답장 버튼 — 메시지 옆 ↩ 클릭 시 입력창 즉시 펼쳐짐
- 👍 반응 버튼
- 메시지는 Redis TTL, 만료 후 자동 삭제
- AI(노잇) 개입 없음 — 사람 간 직접 소통

### 메시지 구조 (Redis)
```json
{
  "id": "uuid",
  "userId": 123,
  "nickname": "닉네임",
  "content": "연차 반차 차이가 뭐예요?",
  "replyToId": null,
  "createdAt": "2026-06-03T10:00:00"
}
```

### API / WebSocket 엔드포인트
| 타입 | 경로 | 설명 |
|---|---|---|
| STOMP Subscribe | `/topic/flash-chat` | 채널 구독 (메시지 수신) |
| STOMP Send | `/app/flash-chat/send` | 메시지 전송 |
| STOMP Send | `/app/flash-chat/react` | 👍 반응 |
| REST GET | `/flash-chat/messages` | 현재 활성 메시지 목록 (최초 진입 시) |

### 유저 설정
| 설정 | 설명 |
|---|---|
| 알림 on/off | 새 메시지 뱃지 및 푸시 알림 수신 여부 토글 |

### SYSTEM_ADMIN — Flash Chat 관리
| 설정 | 설명 | 기본값 |
|---|---|---|
| 메시지 TTL | 메시지 자동 삭제까지의 시간. Redis TTL로 적용. | 600초 (10분) |
| 도배 방지 쿨다운 | 같은 유저가 연속 전송 가능한 최소 간격. | 3초 |
| 메시지 강제 삭제 | 관리자가 특정 메시지 즉시 삭제. `admin_logs`에 기록. | — |
| 금지어 목록 | 포함 시 전송 차단되는 단어 목록. 관리자 페이지에서 편집. | — |

### 미결 사항 (회의 시 논의)
- 메시지 최대 보존 개수 (스크롤 overflow 방지)

## 2. 모바일 지원

### 개요
기존 React 앱에 반응형 + PWA + 사진 첨부 추가.
"현장에서 바로 사진 찍어 티켓 요청" 시나리오 지원.

### 구성
| 항목 | 방법 |
|---|---|
| 반응형 UI | Tailwind CSS 반응형 클래스 (`sm:`, `md:`) 적용 |
| 앱 래핑 | Capacitor + Vue — 기존 웹앱을 iOS/Android 앱으로 패키징 |
| iOS 배포 | TestFlight (데모용) |
| Android 배포 | APK 직접 설치 (데모용) |
| 사진 첨부 | Capacitor Camera 플러그인 또는 `<input type="file" accept="image/*" capture>` |
| 파일 저장 | 백엔드 multipart 업로드 API + 로컬 or S3 |

### 추가 API
| Method | Path | 설명 |
|---|---|---|
| POST | `/attachments` | 이미지 업로드, `attachment_id` 반환 |
| GET | `/attachments/{id}` | 이미지 조회 |

### 미결 사항 (회의 시 논의)
- 이미지 저장소: 로컬 파일시스템 vs S3 (배포 환경에 따라)
- 프론트 담당자(황희수)와 반응형 작업 범위 조율 필요

# 파트 2 — AI 파트 의사결정 및 설계 명세

## AI 도입 배경 — RAG vs QLoRA

단순 RAG 또는 단순 파인튜닝 중 어느 쪽으로 가야 하는지 비교 검토한 결과, 두 접근의 역할을 분리하는 방향으로 결정했다.

| 방식 | 강점 | 한계 |
|---|---|---|
| RAG만 | 최신 문서 기반 답변, 출처 명시, hallucination 억제 | 사내 특화 행동 패턴 학습 불가 (거절 방식, 워키 유도, 출처 표시 형식 등) |
| QLoRA만 | 행동 패턴 학습, 사내 어투·규칙 내재화 | 지식을 학습시키면 hallucination 발생. 완전 제거에 수천 건 이상 데이터 필요 |
| **RAG + QLoRA ✓** | 지식은 RAG로 정확하게, 행동은 QLoRA로 학습 — 각자의 강점만 활용 | 구조가 다소 복잡하나 역할이 명확히 분리됨 |

### 최종 결정

> **RAG + QLoRA 혼합 아키텍처로 확정.**
> RAG가 지식 정확성을 담당하고, QLoRA가 사내 특화 행동 패턴을 담당한다.
> 핵심 원칙: QLoRA에게 "무엇을 알아야 하는가"를 가르치지 않고, "어떻게 행동해야 하는가"만 가르친다.

---

## 섹션 1. QLoRA 실험 결과

### 실험 환경
- 하드웨어: MacBook Pro M5 Pro 48GB
- 프레임워크: MLX-LM (Apple Silicon 최적화)
- 베이스 모델: Llama-3.2-3B-Instruct (4bit 양자화)

### 실험 결과
| 실험 | 조건 | Val Loss | 소요 / 메모리 |
|---|---|---|---|
| 1차 | 데이터 15개, iters 100 | 3.889 → **0.742** | 약 2분 / 2.9GB |
| 2차 | 데이터 50개, iters 300 | 3.889 → **0.360** (iter 200 최저) | 약 5분 / 2.9GB |

> iter 200 이후 소폭 상승 → 과적합 시작 구간 확인

### Before / After 비교
테스트 질문: "육아휴직 얼마나 쓸 수 있어?" (참고 문서 없음 상황)

| | 기본 모델 | 파인튜닝 후 |
|---|---|---|
| 첫 문장 | ✗ 어색한 거부 | ✓ "참고 문서에 육아휴직 관련 내용이 없어 답변드리기 어렵습니다." |
| 이후 내용 | ✗ 날짜 지어냄 (hallucination) | △ 일부 hallucination 여전히 발생 |
| 워키 유도 | ✗ 없음 | △ 시도하나 완전하지 않음 |

### 실험 결론
- 소량 데이터로도 **행동 패턴 학습은 가능**함을 확인
- QLoRA로 지식까지 가르치려 하면 hallucination 위험이 생김
- → **지식은 RAG, 행동은 QLoRA**로 역할 분리가 최적

## 섹션 2. 최종 AI 아키텍처

```
사용자 질문
    ↓
[QLoRA 파인튜닝된 모델] — 행동 패턴 담당
    ↓
RAG로 문서 검색 — 지식 내용 주입
    ↓
최종 답변 + 출처 + 날짜 표시
```

### 단계별 고도화 플랜
| 단계 | 시점 | 내용 |
|---|---|---|
| Phase 1 | 납품 초기 | 매뉴얼 + 워키 RAG 기반 답변. 행동 패턴은 시스템 프롬프트로 제어. 출처 + 날짜 표시. |
| Phase 2 | 데이터 축적 후 | APPROVED knowledge_candidates 누적 → 임계값 초과 시 QLoRA 자동 파인튜닝. 사내 환경 최적화. |

> 쓸수록 해당 기업에 맞게 고도화되는 구조. B2B 솔루션으로서 장기적인 경쟁력이 생긴다.

## 섹션 3. 프롬프트 관리

챗봇 시스템 프롬프트를 두 영역으로 분리해서 관리한다.

| 영역 | 역할 | 수정 시 재학습 | 편집 권한 |
|---|---|---|---|
| `base_system` | QLoRA 학습 기준 프롬프트. 핵심 행동 규칙 정의. | 필요 (자동 트리거) | SYSTEM_ADMIN |
| `admin_context` | 회사/부서 맞춤 설정. 부서별 역할, 특화 지침. | 불필요 (즉시 반영) | SYSTEM_ADMIN |

> ⚠️ base_system 저장 시: "AI 재학습이 자동으로 시작됩니다. 완료까지 약 10분 소요됩니다."

### 런타임 적용 구조
```
final_system_prompt = base_system + "\n\n" + admin_context
```
매 요청마다 두 영역을 합쳐서 LLM에 전달한다.

## 섹션 5. 파인튜닝 자동화 파이프라인

### 학습 데이터 소스
| 소스 | 조건 |
|---|---|
| `worki_answers` | `official = true` |
| `knowledge_candidates` | `status = APPROVED` + 유효기간 이내 |

> 유효한 학습 데이터가 임계값(기본: 100건) 초과 시 자동 실행. 임계값도 관리자 페이지에서 설정 가능.

### 전체 흐름
```
TEAM_ADMIN 공식 답변 작성 (official = true)
            ↓
knowledge_candidates 등록 → APPROVED 상태
            ↓
유효 데이터 임계값 초과
            ↓
JSONL 자동 변환
            ↓
mlx_lm.lora 백그라운드 실행
            ↓
완료 → 어댑터 자동 교체 → 챗봇 즉시 반영
```
