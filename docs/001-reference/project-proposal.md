## 1. 기획 개요

Workipedia는 사내 구성원이 회사 규정, 업무 매뉴얼, 동료의 경험 지식을 빠르게 찾고, 실제 처리가 필요한 업무 요청을 담당 부서 티켓으로 연결하는 **질문/요청 기반 사내 지식화 플랫폼**이다.

핵심은 사용자의 입력을 `질문`과 `요청`으로 분리하는 것이다.

```text
질문 = 기존 지식을 찾는 기능
요청 = 새로운 지식 후보를 만드는 기능
```

질문은 챗봇이 매뉴얼과 워키를 검색해 출처 기반 답변을 제공하고, 요청은 담당 부서 티켓으로 처리된다. 처리 완료된 요청은 개인 내용을 일반화해 워키 지식 후보로 전환한다.

## 2. 문제 정의

현재 사내 지식과 업무 요청은 여러 채널에 흩어져 있다.

| 문제 | 설명 |
|---|---|
| 지식 탐색 비용 증가 | 규정, 매뉴얼, FAQ가 분산되어 신입이나 비전문가가 필요한 정보를 찾기 어렵다. |
| 반복 문의 누적 | 같은 질문이 메신저, 메일, 구두 문의로 반복되지만 조직 지식으로 축적되지 않는다. |
| 처리 이력 부재 | 업무 요청이 개인 메일함에 남아 담당 부서, 처리 상태, 답변 이력이 구조화되지 않는다. |
| 지식 재사용 부족 | 한 번 처리한 요청이 다음 사람의 질문을 줄이는 자료로 전환되지 않는다. |
| AI 답변 신뢰 부족 | 일반 챗봇 답변은 근거가 없으면 실제 업무에 바로 쓰기 어렵다. |

따라서 본 서비스는 단순 챗봇이 아니라, **질문을 해결하고 요청을 처리하며 처리 결과를 다시 지식으로 축적하는 순환 구조**를 목표로 한다.

## 3. 기획 의도

Workipedia의 기획 의도는 다음 세 가지다.

1. 사내 구성원이 필요한 정보를 빠르게 찾도록 돕는다.
2. 실제 처리가 필요한 업무 요청을 담당 부서 티켓으로 구조화한다.
3. 처리된 요청을 지식화하여 다음 반복 문의를 줄인다.

서비스의 최종 순환 구조는 다음과 같다.

```text
워키/매뉴얼 지식
→ 사용자 질문
→ 챗봇 출처 기반 답변
→ 해결 불가 또는 실제 처리 필요
→ 요청 티켓 발행
→ 담당 부서 처리
→ 처리 답변 지식화
→ 워키 강화
→ 다음 질문 자동 해결
```

## 4. 핵심 사용자

| 사용자 | 니즈 | 제공 가치 |
|---|---|---|
| 일반 사원 | 사내 규정과 업무 절차를 빠르게 알고 싶다. | 챗봇이 매뉴얼과 워키를 근거로 답변한다. |
| 요청자 | 실제 업무 처리나 공식 확인이 필요하다. | 요청 티켓으로 담당 부서 처리 상태를 확인한다. |
| 담당 팀원 | 들어온 요청을 누락 없이 처리하고 싶다. | 담당 티켓 큐와 배정 기능으로 업무를 관리한다. |
| 팀 관리자 | 자기 팀 티켓을 확인하고 팀원에게 분배해야 한다. | 팀 큐, 팀원 배정, 지식화 승인 기능을 제공한다. |
| 전체 관리자 | 자동 배정 실패와 운영 흐름을 관리해야 한다. | 공통 접수 큐와 운영 지표를 제공한다. |

## 5. 서비스 구조

```text
메인 진입
├─ 질문
│  └─ 챗봇 검색
│     ├─ 매뉴얼 근거 답변
│     ├─ 워키 근거 답변
│     └─ 답변 실패/불충분 시 요청 티켓 전환
│
└─ 요청
   └─ 티켓 발행
      ├─ 라우팅 신뢰도 높음 → 담당 부서 자동 배정
      └─ 라우팅 신뢰도 낮음 → 공통 접수 큐
         └─ 전체 관리자 수동 배정
```

질문은 정보 탐색 중심이고, 요청은 실제 업무 처리 중심이다. 두 흐름은 분리되어 있지만, 챗봇 답변이 없거나 공식 처리가 필요한 경우 질문에서 요청으로 전환될 수 있다.

## 6. 핵심 기능

### 6.1 질문/챗봇

- 사용자가 자연어로 사내 규정, 업무 절차, FAQ를 질문한다.
- 챗봇은 매뉴얼과 워키를 검색해 답변한다.
- 답변에는 반드시 매뉴얼/워키 출처가 포함된다.
- 근거가 부족하면 그럴듯한 답변을 만들지 않고 요청 티켓 전환을 안내한다.

### 6.2 요청 티켓

- 사용자는 실제 처리가 필요한 요청을 티켓으로 발행한다.
- AI가 담당 부서와 라우팅 신뢰도를 산출한다.
- 신뢰도가 높으면 담당 부서 큐로 자동 배정한다.
- 신뢰도가 낮으면 공통 접수 큐로 이동한다.
- 팀 관리자는 자기 팀 티켓을 확인하고 팀원에게 분배한다.

### 6.3 공통 접수 큐

- 자동 배정이 애매한 티켓을 모아두는 운영 큐다.
- 전체 관리자가 내용을 확인하고 담당 부서를 수동 배정한다.
- 팀 관리자의 이관 요청도 다른 부서로 바로 이동하지 않고 공통 접수 큐로 돌아간다.

### 6.4 지식화

- 처리 완료된 티켓은 지식화 후보가 될 수 있다.
- 담당자가 지식화 후보 등록 여부를 선택한다.
- AI가 개인적인 내용을 제거하고 일반 절차 형태로 요약한다.
- 팀 관리자가 검수한 뒤 워키에 반영한다.

### 6.5 포인트/뱃지/ESG

- 질문, 답변, 채택, 지식화 활동에 포인트를 부여한다.
- 참여를 유도하기 위해 기본 뱃지를 제공한다.
- 지식 공유량, 지식화 전환 건수, 예상 절감 시간 등을 ESG 지표 카드로 보여준다.

## 7. 대표 시나리오

### 7.1 질문 → 챗봇 → 답변

```text
사용자가 "연차 신청은 어디서 하나요?" 질문
→ 챗봇이 매뉴얼과 워키 검색
→ 매뉴얼 답변과 워키 답변을 출처와 함께 제공
→ 사용자가 링크를 확인하고 해결
```

### 7.2 질문 → 챗봇 → 티켓 발행

```text
사용자가 "해외 출장 중 의료비 처리는 어떻게 하나요?" 질문
→ 챗봇이 충분한 근거를 찾지 못함
→ "담당 부서에 요청하기" 안내
→ 질문 내용이 요청 초안으로 전환
→ 요청 티켓 발행
```

### 7.3 요청 → 자동 배정 → 처리 완료

```text
사용자가 "VPN 접속 오류 처리 요청합니다." 요청 작성
→ AI가 IT지원팀으로 자동 배정
→ IT지원팀 팀 관리자가 티켓 확인
→ 담당 팀원에게 분배
→ 팀원이 조치 결과 작성
→ 처리 완료 및 사용자 알림
```

### 7.4 요청 → 처리 완료 → 지식화

```text
사용자가 "노트북 반납 신청합니다." 요청 작성
→ 자산관리팀 자동 배정
→ 담당 팀원이 반납 절차 안내 후 처리 완료
→ 담당자가 지식화 후보 등록
→ AI가 개인 내용을 제거하고 일반 절차로 요약
→ 팀 관리자가 검수
→ 워키에 "노트북 반납 절차" 반영
```

### 7.5 요청 → 공통 접수 큐 → 수동 배정

```text
사용자가 "SSD 추가 장착과 보안씰 처리를 확인받고 싶습니다." 요청 작성
→ AI가 자산관리팀/정보보안팀 사이에서 낮은 신뢰도 산출
→ 공통 접수 큐로 이동
→ 전체 관리자가 담당 부서를 지정
→ 해당 부서 팀 관리자가 팀원에게 분배
→ 처리 완료
```

## 8. 권한 구조

| Role | 주요 기능 |
|---|---|
| `USER` | 질문 작성, 요청 티켓 발행, 본인 티켓 상태 확인, 워키 답변 작성 |
| `TEAM_ADMIN` | 자기 팀 티켓 확인, 팀원 배정, 티켓 이관 요청, 지식화 승인 |
| `SYSTEM_ADMIN` | 공통 접수 큐 관리, 자동 배정 실패 검토, 부서/카테고리 매핑, 운영 지표 확인 |

전체 관리자는 개인 평가자가 아니라 서비스 운영 관리자다. 대시보드는 개인별 실적보다 팀 단위 처리 현황, 공통 접수 큐, 자동 배정 성공률, 지식화 전환 건수 중심으로 제공한다.

## 13. 발표 핵심 메시지

> Workipedia는 단순히 AI가 답변하는 챗봇이 아니라, 질문을 해결하고 요청을 처리하며 처리 결과를 다시 조직 지식으로 축적하는 사내 지식화 플랫폼이다.

> 질문은 기존 지식을 소비하는 흐름이고, 요청은 새로운 지식을 만들어내는 흐름이다.

> 모든 챗봇 답변은 매뉴얼과 워키 출처를 기반으로 하며, 근거가 부족한 경우 요청 티켓으로 전환해 담당 부서의 공식 처리를 받는다.
