# Constitution — Worki-t 프로젝트 헌법

> 문서 유형: Constitution
> 상태: Draft
> 정본 위치: `docs/reference/constitution.md`
>
> 본 문서는 Worki-t 프로젝트의 **불변 원칙(Principles)** 과 **의사결정 기준** 을 정의한다.
> PRD/TRD는 변경되더라도 본 헌법의 원칙에 반하는 변경은 허용되지 않는다.
> 헌법의 수정에는 팀 합의(또는 정의된 운영 위원회 승인)가 필요하다.
>
> 버전: v0.1
> 최종 수정: 2026-05-28

---

## 1. 프로젝트 미션

Worki-t은 **사내 구성원이 신뢰할 수 있는 지식에 가장 빠르게 도달할 수 있는 통로** 가 되어야 한다.
모든 기능 결정은 "이 변경이 사용자가 신뢰할 수 있는 지식에 더 빨리 닿게 하는가?"라는 질문을 통과해야 한다.

---

## 2. 핵심 원칙 (Core Principles)

### Principle 1. 출처 없는 답변은 없다 (Source-First)
- 챗봇이 매뉴얼 및 워키를 근거로 답변할 때는 **반드시 인용·출처 링크를 함께 제공한다.**
- 출처 없이 그럴듯한 답변을 생성하는 것은 본 서비스의 존재 이유에 반한다.
- 출처를 표시할 수 없으면 "답변을 찾지 못했다"고 말하고 워키 등록 흐름으로 안내한다.

### Principle 2. 조직 지식은 개인의 자산이 아니라 조직의 자산이다 (Knowledge Belongs to the Org)
- 일반 사용자는 본인이 등록한 질문에 답변이 작성되기 전까지 질문을 수정 및 삭제할 수 있다.
- 답변·티켓·채택 이력은 보존된다(soft delete만 허용).
- 답변이 작성되었거나 티켓·채택 이력이 생긴 게시글의 수정 및 삭제는 관리자 권한이며, 모든 활동은 `admin_logs`에 기록된다.

### Principle 3. 신뢰성 > 응답 속도 > 비용 (Trust First)
- 의사결정이 충돌할 때 우선순위는 신뢰성 > 응답 속도 > 비용 순서이다.
- 예: LLM 답변 품질이 의심되면 캐시된 그럴듯한 답변보다 "찾지 못함" 응답을 택한다.

### Principle 4. 개인정보는 들어오지도, 나가지도 않는다 (Privacy by Default)
- 입력 단계에서 개인정보(주민번호, 연락처, 계좌 등)는 마스킹 후 저장한다.
- 출력 단계에서 개인정보 유출 가능성이 있는 응답은 거부한다.
- 질문자의 부서명은 기본 비공개이며, 노출 정책은 명시적으로 정의된 경우(예: 담당 부서 공식 답변)에만 허용된다.

### Principle 5. 권한과 책임은 명시적이어야 한다 (Explicit Authority)
- 역할은 `USER`와 `ADMIN` 두 단계로만 모델링한다. 권한을 늘리려는 결정은 헌법 수정 절차에 준한다.
- 관리자 작업은 모두 추적 가능해야 한다(admin_logs).
- 자동화된 의사결정(예: 티켓 부서 배정)도 사후 감사가 가능해야 한다.

### Principle 6. 참여는 보상한다, 그러나 보상을 위해 품질을 희생하지 않는다 (Incentivize, Don't Game)
- 포인트·뱃지·ESG 시각화는 자발적 참여를 유도하기 위한 장치이다.
- 다만 보상을 위해 저품질 답변, 어뷰징성 활동이 늘어나면 정책은 즉시 재조정된다.
- 정량(좋아요 수, 답변 수)과 정성(채택, 인용) 지표를 함께 본다.

### Principle 7. 챗봇 → 워키 → 티켓의 에스컬레이션 경로는 깨지지 않는다 (Always an Escalation Path)
- 사용자는 챗봇에 막혀서는 안 된다. 챗봇이 답변하지 못한 질문은 워키 게시판에 등록할 수 있으며, 이 경우 티켓이 발행된다.
- 어떤 흐름도 막다른 길(dead end)이어서는 안 된다.

### Principle 8. ESG는 마케팅이 아니라 측정 가능한 지표다 (ESG as Metric)
- ESG-S: 지식 공유량, 신입·비전문가 접근성 개선 지표.
- ESG-G: 매뉴얼 갱신 빈도, 관리자 검증 건수, 답변 이력 보존율.
- ESG 시각화는 정량 지표에 근거해야 하며, 시각적 장식만으로 ESG를 주장하지 않는다.

## 3. 기술 원칙 (Technical Principles)

### T1. 데이터는 정본(Source of Truth)이 하나여야 한다
- 매뉴얼은 `manuals` 테이블이 정본, 워키는 `worki_questions`/`worki_answers`가 정본.
- Vector Store는 검색 인덱스일 뿐 정본이 아니다. 손실되어도 정본에서 재구축 가능해야 한다.

### T2. RAG의 모든 답변은 추적 가능해야 한다
- 챗봇 응답은 어떤 chunk를 참조했는지 `chatbot_messages.references`에 JSON으로 남긴다.
- 사후 감사·디버깅·품질 개선에 활용된다.

### T3. 변경은 마이그레이션과 함께 한다
- 스키마 변경은 Flyway/Liquibase 등 마이그레이션 도구로만 수행한다.
- 운영 DB에 수동 DDL을 직접 가하지 않는다.

### T4. 비밀(Secret)은 코드에 들어가지 않는다
- API 키, 비밀번호, JWT 시크릿은 시크릿 매니저로 관리한다.
- 저장소 커밋 전 시크릿 스캐너로 검사한다.

### T5. 외부 의존성은 격리되어야 한다
- LLM·Embedding 등 외부 API 호출은 추상화 계층(Adapter)을 통해 사용한다.
- 단일 벤더 락인을 피하고, 장애 시 fallback 경로를 가진다.

### T6. 배치와 실시간은 분리한다
- 임베딩 갱신, 통계 집계 등은 배치 워커가 담당한다.
- 사용자 요청 처리 경로에 무거운 배치 작업을 끼워 넣지 않는다.

## 4. 프로세스 원칙 (Process Principles)

### P1. 정책 변경은 문서가 먼저, 코드는 다음이다
- 정책(예: 닉네임 중복, 워키 우선순위, 관리자 범위)이 변경되면 정책 기술서 → PRD → TRD → 코드 순으로 반영한다.
- 정책 미확정 상태로 코드가 먼저 나가지 않는다.

### P2. 미확정 정책은 명시적으로 표시한다
- PRD/TRD/정책 기술서에서 "미확정" 항목은 반드시 표시되어야 한다.
- 미확정 항목은 추측이 아니라 결정으로 전환되어야 한다.

### P3. 결정은 기록한다 (ADR)
- 중요한 기술 결정(예: Vector Store 선택, 인증 방식)은 Architecture Decision Record로 남긴다.
- "왜 이걸 안 골랐는가"까지 남긴다.

### P4. 코드 리뷰는 최소 1인 (보안·DB·외부 의존 변경은 2인)
- 인증·권한·개인정보·DB 마이그레이션·외부 API 변경은 2인 이상 리뷰가 필수다.

### P5. 사용자 데이터를 다루는 기능은 테스트가 있다
- 권한 검사, 개인정보 마스킹, 삭제 정책에는 자동화 테스트가 동반된다.

---

## 5. 충돌 해결 규칙

본 헌법의 원칙들이 충돌할 때:
1. **사용자 안전·개인정보(Principle 4) > 모든 것**
2. **데이터 무결성·조직 지식 보존(Principle 2) > 사용성 편의**
3. **신뢰성(Principle 3) > 응답 속도 > 비용**
4. 위 규칙으로도 해결되지 않으면 운영 위원회(또는 PO + Tech Lead) 합의로 결정하고 ADR을 남긴다.

---

## 6. 헌법 수정 절차

1. 수정 제안자는 변경 사유, 영향 범위, 대안을 정리한다.
2. PO 또는 Tech Lead의 검토를 받는다.
3. 팀(운영 위원회)의 합의로 확정한다.
4. 헌법 버전을 올리고, 변경 이력을 본 문서 하단에 기록한다.
