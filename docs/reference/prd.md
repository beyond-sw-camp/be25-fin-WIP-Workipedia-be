## 1. 문서 개요

### 1.1 목적
Workipedia는 사내 구성원이 회사 규정·매뉴얼·동료의 경험 지식을 한 곳에서 검색하고, 실제 처리가 필요한 업무 요청을 담당 부서 티켓으로 연결하는 **질문/요청 기반 사내 지식화 플랫폼**이다. AI 어시스턴트 **노잇(KnowIT)** 이 매뉴얼과 워키를 근거로 답변하고, 해결되지 않은 요청은 처리 후 승인된 지식 데이터로 축적될 수 있다. 본 문서는 Workipedia의 제품 비전, 사용자, 핵심 기능, 우선순위, 성공 지표를 정의한다.

### 1.2 적용 범위
본 PRD는 MVP 범위(사용자 계정, RAG 기반 챗봇, 워키, 요청 티켓/부서 라우팅, 승인 지식 데이터, Tool Integration, 포인트/ESG 등급, 알림, FAQ, ESG 지표 카드, 관리자 대시보드, Flash Chat, 모바일/사진 첨부)를 대상으로 한다.

## 2. 제품 비전 및 배경

### 2.1 문제 정의
- 사내 규정·업무 매뉴얼이 여러 시스템에 흩어져 있어 신입·비전문가가 필요한 정보를 찾기 어렵다.
- 동료에게 같은 질문이 반복적으로 발생하며, 답변이 개인 메신저에 머물러 조직 자산으로 축적되지 않는다.
- 부서 간 공식적인 업무 요청 채널이 메일·메신저에 흩어져 처리 이력과 책임 소재가 남지 않는다.
- 처리된 요청이 개인 메일함에만 남아, 반복 요청을 줄이는 조직 지식으로 전환되지 않는다.

### 2.2 솔루션
1. **질문** — 기존 지식을 찾는 입력이다. 챗봇이 매뉴얼과 워키를 고객사별 Embedding provider와 Qdrant로 검색하여 출처와 함께 답변한다.
2. **요청** — 실제 처리가 필요한 입력이다. 요청은 티켓으로 발행되고 담당 부서 큐에서 처리된다.
3. **티켓 라우팅** — AI가 담당 부서와 신뢰도를 산출한다. 신뢰도가 높으면 자동 배정하고, 낮으면 공통 접수 큐로 보낸다.
4. **지식화** — 처리 완료된 요청 답변은 개인 정보를 제거하고 TEAM_ADMIN 승인 후 지식 데이터로 저장된다.
5. **포인트/ESG 등급/지표** — 지식 공유 참여와 지식화 효과를 보상하고, ESG-S/G 지표로 시각화한다.
6. **Flash Chat** — 공식 티켓이나 워키로 남기기 전 가벼운 질문을 전사 공개 실시간 채팅에서 빠르게 해결한다.

### 2.3 제품 핵심 가치
- **출처 기반 신뢰성**: 매뉴얼·워키 기반 답변에는 항상 근거 문서·인용을 제공한다.
- **질문/요청 분리**: 정보 탐색은 챗봇과 워키로 가볍게 해결하고, 실제 처리는 공식 티켓으로 관리한다.
- **조직 지식 자산화**: 처리된 요청은 개인 사례를 일반화해 워키 지식 후보로 축적한다.
- **참여 보상**: 포인트, ESG 등급, ESG 시각화로 자발적 참여를 유도한다.

## 3. 대상 사용자

### 3.1 사용자 분류
| 구분 | 권한(role) | 설명 |
|---|---|---|
| 일반 사용자 | USER | 사내 구성원. 질문, 요청 티켓 발행, 답변 작성, 본인 티켓 상태 확인 가능. |
| 팀 관리자 | TEAM_ADMIN | 자기 팀 티켓 큐 확인, 팀원 배정, 답변 관리, 지식화 승인 가능. |
| 전체 관리자 | SYSTEM_ADMIN | 공통 접수 큐, 자동 배정 실패, 부서 R&R, AI Tool, 수기 지식과 전체 운영 지표 관리. |

### 3.2 페르소나 (요약)
- **신입사원 김OO** — 사내 규정을 검색하기 어려움. 챗봇으로 즉시 답을 얻고, 모를 때는 워키에 질문.
- **시니어 이OO** — 동료의 반복 질문에 피로감. 워키에 한 번 답하면 이후 챗봇이 그 답을 재활용해 부담이 줄어듦.
- **자산관리팀 박OO(TEAM_ADMIN)** — 자기 팀 티켓 큐를 확인하고 담당 팀원에게 분배.
- **운영관리자 최OO(SYSTEM_ADMIN)** — 공통 접수 큐와 자동 배정 실패 티켓을 검토하고 운영 지표를 관리.

### 4.3 워키 (WORKI) — 오픈 Q&A 게시판
- 챗봇에서 등록 흐름 진입 시 챗봇 입력 내용을 기반으로 질문 등록 가능
- 질문 상태: WAITING(대기) → 답변 진행 → 채택 완료 (티켓 발행 시 별도 흐름)
- WAITING 상태에서만 질문 수정 가능, **사용자는 직접 삭제 불가** (조직 지식 보존)
- 동일 질문에 채택된 답변이 존재하면 추가 답변 등록 불가
- 좋아요/싫어요 반응 (싫어요 = 답변이 부족하다는 피드백)
- 질문자 부서명 기본 비공개, 닉네임만 노출

### 4.4 매뉴얼 (MANUAL)
- 사내 규정, 인사 규정, 업무 규정 등 **공식 문서** 기반
- SYSTEM_ADMIN만 등록·수정·삭제 가능
- 챗봇이 매뉴얼·워키 기반 답변 시 반드시 인용 정보 포함
- 인용 빈도 기준 인기 매뉴얼 노출 (FAQ_002)

### 4.6 지식화 (KNOWLEDGE)
- 처리 완료된 티켓은 TEAM_ADMIN 승인 후 `knowledge_data`로 저장한다.
- AI가 티켓 답변을 재사용 가능한 지식 초안으로 요약하되, 개인 사례·개인정보·민감 정보를 제거한다.
- TEAM_ADMIN이 지식화 내용을 검수하고 승인한다.
- 지식화된 문서는 이후 챗봇 답변의 워키 근거로 사용될 수 있다.

### 4.7 포인트 · ESG 등급
- 지식 공유 활동(질문, 답변, 채택, 첫 질문 등)에 포인트 부여
- 누적 ESG 점수 기준 등급 산정
- 포인트 누적 순위에 따라 랭킹 제공
- 첫 질문 등록 시 20P 등 튜토리얼성 보상 포함 (TUTORIAL_004)

### 4.8 알림 (NOTIFICATION)
- 채택, 답변 등록, 티켓 상태 변경 등 이벤트 발생 시 사용자에게 알림
- 알림 클릭 시 해당 게시글 또는 매뉴얼로 이동
- 모두 읽음 처리, 개별 알림 삭제 지원

### 4.9 FAQ / 메인
- FAQ_001: 인용수 많은 매뉴얼 + 좋아요 많은 워키를 메인에 요약
- FAQ_002: 인기 매뉴얼 (제목, 소제목, 개정일)
- FAQ_003: 인기 워키 (좋아요 기준)

### 4.10 ESG 시각화
- 사용자의 지식 공유 활동을 ESG 지표 카드로 시각화
- 고도화된 그래픽 시각화보다 측정 가능한 지표(지식 공유량, 예상 절감 시간, 출처 포함 답변 비율)를 우선한다.
- 챗봇 해결 건수, 예상 절감 시간, 지식화 전환 건수, 지식화 문서 재사용 횟수, 자동 배정 성공률을 포함한다.
- **ESG-S**: 구성원 간 지식 공유, 신입·비전문가 정보 접근성 향상
- **ESG-G**: 공식 규정 근거 제공, 답변 이력 관리, 관리자 검증, 투명한 지식 관리 체계

### 4.11 관리자 대시보드
- TEAM_ADMIN 메뉴: 자기 팀 티켓 큐, 담당자 미배정 티켓, 오래 방치된 티켓, 지식화 승인
- SYSTEM_ADMIN 메뉴: 공통 접수 큐, 자동 배정 실패 티켓, 부서 R&R, 전체 운영 지표
- AI 관리 메뉴: 프롬프트 관리, API Tool 관리, 승인된 DB Query Tool 활성화, 수기 지식 관리
- 전체 대시보드는 개인 평가가 아니라 서비스 운영 개선 목적이며, 개인별 성과보다 팀 단위 통계와 운영 메타데이터 중심으로 제공한다.
- 워키 게시글 삭제 권한은 관리자 전용이며, 모든 관리자 작업은 admin_log 테이블에 기록한다.

### 4.13 모바일/사진 첨부
- 모바일 환경에서 로그인, 챗봇, 티켓 발행, 내 티켓 확인, 알림 확인 흐름이 깨지지 않아야 한다.
- 현장 사용자는 티켓 발행 시 사진을 촬영하거나 파일을 선택해 첨부할 수 있다.
- 파일은 BE DB가 아닌 Object Storage에 저장하며, 고객사 배포 환경에 따라 R2/S3/MinIO provider를 선택한다.
- 저장소 provider는 배포 설정으로 결정하며 관리자 화면에서 변경하지 않는다.
- 데모 목적의 모바일 앱 래핑은 PWA 또는 Capacitor를 검토한다.
- 외부 페이지에 챗봇 컴포넌트를 스크립트로 주입하는 경우 허용 도메인 정책을 적용한다.

## 5. 비기능 요구사항 (요약)

| 항목 | 요구사항 |
|---|---|
| 보안 | BE RDB 암호화 저장, 사용자에게 반환하는 AI 응답 마스킹, 비밀정보 로그 기록 금지 |
| 신뢰성 | 매뉴얼·워키 기반 답변은 출처·인용 필수 |
| 데이터 보존 | 사용자는 질문 자체 삭제 불가, 답변·티켓·지식화 이력 보존 |
| 임시 데이터 | Flash Chat 메시지는 Redis TTL로 자동 삭제하며 영구 보존하지 않음 |
| 권한 | USER / TEAM_ADMIN / SYSTEM_ADMIN 3단계 권한 모델, 관리자 작업 로그 |
| 갱신 주기 | 워키 임베딩은 1일 1회 배치 갱신 |

## 6. 사용자 시나리오 (요약 플로우)

1. **질문 → 챗봇 → 답변**
   - 사용자가 질문 입력 → 챗봇이 매뉴얼·워키 검색 → 출처 포함 답변 → 해결

2. **질문 → 챗봇 → 요청 티켓**
   - 챗봇 답변 없음/불충분/공식 확인 필요 → 질문 내용이 요청 초안으로 전환 → 티켓 발행

3. **요청 → 자동 배정 → 처리 완료**
   - 사용자가 요청 작성 → AI가 라우팅 신뢰도 산출 → 담당 부서 큐 자동 배정 → TEAM_ADMIN이 팀원에게 분배 → 팀원이 처리 완료

4. **요청 → 처리 완료 → 지식화**
   - 처리 완료 티켓 → AI가 개인 내용을 일반화 → TEAM_ADMIN 승인 → 지식 데이터 저장 및 RAG 반영

5. **요청 → 공통 접수 큐 → 수동 배정**
   - 라우팅 신뢰도 낮음 → 공통 접수 큐 이동 → SYSTEM_ADMIN이 담당 부서 지정 → 해당 부서 TEAM_ADMIN이 팀원 분배 → 처리 완료

6. **부서 확인 → 이관 요청 → 공통 접수 큐 → 재배정**
   - TEAM_ADMIN이 자기 팀 업무가 아니라고 판단 → 이관 요청 → 티켓이 공통 접수 큐로 이동 → SYSTEM_ADMIN이 담당 부서 재배정

- Object Storage의 orphan object 정리·보존 기간 정책

## 9. 발표 핵심 메시지

> Workipedia는 단순히 AI가 답변하는 챗봇이 아니라, 질문을 해결하고 요청을 처리하며 처리 결과를 다시 조직 지식으로 축적하는 사내 지식화 플랫폼이다.

> 질문은 기존 지식을 소비하는 흐름이고, 요청은 새로운 지식을 만들어내는 흐름이다.

> 모든 챗봇 답변은 매뉴얼과 워키 출처를 기반으로 하며, 근거가 부족한 경우 요청 티켓으로 전환해 담당 부서의 공식 처리를 받는다.

## 11. AI 서버 요구사항

> BE는 업무 데이터와 동기화 상태의 정본을 관리하고, AI 서버는 LLM·Embedding·Qdrant 기반 추론과 검색을 담당한다.

### 11.1 RAG와 챗봇

| ID | 요구사항 | 우선순위 |
|---|---|---|
| AI_RAG_001 | 매뉴얼 → 워키 → 승인 지식·수기 지식 → Tool Calling 순서로 폴백한다 | 상 |
| AI_RAG_002 | Vector Search 후보를 Cross-Encoder로 재정렬하고 근거가 검증된 답변만 반환한다 | 상 |
| AI_RAG_003 | 응답 출처에 `sourceType`, `sourceId`, `chunkIndex`, 제목과 점수를 포함한다 | 상 |
| AI_RAG_004 | BE가 전달한 최근 대화로 검색 쿼리를 재작성하되, 최종 답변에는 현재 질문 원문과 대화 이력을 사용한다 | 중 |
| AI_RAG_005 | 사용자에게 반환하는 최종 LLM 응답에 민감정보 마스킹을 적용한다 | 상 |

### 11.2 티켓 라우팅과 지식 동기화

| ID | 요구사항 | 우선순위 |
|---|---|---|
| AI_ROUTE_001 | 티켓 제목·내용을 부서 R&R과 승인 사례에 비교해 후보 부서 Top 3와 점수를 반환한다 | 상 |
| AI_ROUTE_002 | AI는 `AUTO_ASSIGNED` 또는 `COMMON_QUEUE` 결정을 반환하고 BE가 실제 티켓 상태를 반영한다 | 상 |
| AI_SYNC_001 | 부서 R&R과 승인 사례를 `sourceType + sourceId` 기반 deterministic point ID로 멱등 upsert한다 | 상 |
| AI_SYNC_002 | BE는 `ai_sync_jobs`로 동일 source 작업 순서, 재시도와 최종 동기화 상태를 관리한다 | 상 |

### 11.3 Tool Calling과 운영 경계
