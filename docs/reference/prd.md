## 1. 문서 개요

### 1.1 목적
KnowIT은 사내 구성원이 회사 규정·매뉴얼·동료의 경험 지식을 한 곳에서 검색·질문·답변할 수 있도록 돕는 **AI 기반 사내 지식 공유 플랫폼**이다. 본 문서는 KnowIT의 제품 비전, 사용자, 핵심 기능, 우선순위, 성공 지표를 정의한다.

### 1.2 적용 범위
본 PRD는 MVP 범위(사용자 계정, 로컬 임베딩 기반 챗봇, 워키, 티켓/이관, 포인트/뱃지, 알림, FAQ, ESG 지표 카드, 관리자 대시보드)를 대상으로 한다.

## 2. 제품 비전 및 배경

### 2.1 문제 정의
- 사내 규정·업무 매뉴얼이 여러 시스템에 흩어져 있어 신입·비전문가가 필요한 정보를 찾기 어렵다.
- 동료에게 같은 질문이 반복적으로 발생하며, 답변이 개인 메신저에 머물러 조직 자산으로 축적되지 않는다.
- 부서 간 공식적인 질의응답 채널이 부재해, 책임 소재가 불분명한 정보가 유통된다.

### 2.2 솔루션
1. **챗봇(RAG)** — 매뉴얼과 워키를 로컬 임베딩 기반으로 검색하여 자연어 질문에 출처와 함께 답변한다.
2. **워키(Wiki + Q&A) 게시판** — 챗봇이 답하지 못한 질문을 오픈 게시판에 등록해 동료가 답할 수 있다.
3. **티켓** — 공식 답변이 필요한 질문은 AI가 담당 부서로 자동 배정한다.
4. **포인트/뱃지/ESG** — 지식 공유 참여를 보상하고, ESG-S/G 지표로 시각화한다.

### 2.3 제품 핵심 가치
- **출처 기반 신뢰성**: 매뉴얼 기반 답변에는 항상 근거 문서·인용을 제공한다.
- **조직 지식 자산화**: 일반 사용자는 질문을 직접 삭제할 수 없으며, 모든 Q&A 이력은 조직 지식으로 보존된다.
- **참여 보상**: 포인트, 뱃지, ESG 시각화로 자발적 참여를 유도한다.

## 3. 대상 사용자

### 3.1 사용자 분류
| 구분 | 권한(role) | 설명 |
|---|---|---|
| 일반 사용자 | USER | 사내 구성원. 챗봇 사용, 질문 등록, 답변 작성, 반응(좋아요/싫어요) 가능. 게시글 삭제 불가. |
| 관리자 | ADMIN | 인사팀 또는 운영 담당 부서. 사용자/워키/매뉴얼/티켓 관리 권한 보유. |
| 담당 부서원 | USER (특정 부서) | 티켓이 배정된 부서의 구성원. 부서 티켓 처리 및 공식 답변 작성. |

### 3.2 페르소나 (요약)
- **신입사원 김OO** — 사내 규정을 검색하기 어려움. 챗봇으로 즉시 답을 얻고, 모를 때는 워키에 질문.
- **시니어 이OO** — 동료의 반복 질문에 피로감. 워키에 한 번 답하면 이후 챗봇이 그 답을 재활용해 부담이 줄어듦.
- **인사팀 박OO(ADMIN)** — 미답변 질문·티켓 처리 현황을 대시보드로 모니터링.
- **법무팀 최OO(담당 부서원)** — 본인 부서에 배정된 티켓에 공식 답변을 작성.

## 4. 핵심 기능 요구사항

### 4.1 사용자 계정 (AUTH)
| ID | 기능 | 우선순위 |
|---|---|---|
| AUTH_001 | 사번/부서명/회사 이메일/비밀번호 기반 회원가입 (비밀번호 8자 이상 영문+숫자) | 상 |
| AUTH_002 | 시스템 자동 닉네임 생성 | 상 |
| AUTH_003 | 사번 기반 로그인 | 상 |
| AUTH_004 | 관리자에 의한 계정 비활성화 (퇴사자 차단) | 상 |

### 4.2 챗봇 (KNOIT) — RAG 기반
| ID | 기능 | 우선순위 |
|---|---|---|
| KNOIT_001 | 자연어로 매뉴얼·워키에 대한 질문 입력 | 상 |
| KNOIT_002 | 매뉴얼·워키 검색 후 답변 생성 (RAG) | 상 |
| KNOIT_003 | 답변에 출처 매뉴얼/워키 링크 제공 | 상 |
| KNOIT_004 | 질문·답변을 메시지 단위로 저장 (대화 세션) | 상 |
| KNOIT_005 | 답변 실패 또는 사용자 불만족 시 워키 질문 등록 흐름으로 연결 | 상 |
| KNOIT_006 | 신규/수정 워키·매뉴얼 임베딩 업데이트 | 중 |
| KNOIT_007 | 개인정보 마스킹 수집 | 상 |
| KNOIT_008 | 개인정보 유출 가능성이 있는 질문 답변 거부 | 상 |

### 4.3 워키 (WORKI) — 오픈 Q&A 게시판
- 챗봇에서 등록 흐름 진입 시 챗봇 입력 내용을 기반으로 질문 등록 가능
- 질문 상태: WAITING(대기) → 답변 진행 → 채택 완료 (티켓 발행 시 별도 흐름)
- WAITING 상태에서만 질문 수정 가능, **사용자는 직접 삭제 불가** (조직 지식 보존)
- 동일 질문에 채택된 답변이 존재하면 추가 답변 등록 불가
- 좋아요/싫어요 반응 (싫어요 = 답변이 부족하다는 피드백)
- 질문자 부서명 기본 비공개, 닉네임만 노출

### 4.4 매뉴얼 (MANUAL)
- 사내 규정, 인사 규정, 업무 규정 등 **공식 문서** 기반
- 관리자만 등록·수정·삭제 가능
- 챗봇이 매뉴얼 기반 답변 시 반드시 인용 정보 포함
- 인용 빈도 기준 인기 매뉴얼 노출 (FAQ_002)

### 4.5 티켓 (TICKET)
| ID | 기능 | 우선순위 |
|---|---|---|
| TICKET_001 | AI가 질문 내용을 분석해 적절한 담당 부서에 티켓 자동 배정 | 상 |
| TICKET_002 | 사용자: 본인 질문에 발행된 티켓을 마이페이지에서 조회 | 상 |
| TICKET_003 | 담당 부서원: 배정 티켓 확인 | 상 |
| TICKET_004 | 시스템: 처리대기/이관 요청/처리완료/반려/삭제 상태 전이 | 상 |
| TICKET_005 | 담당 부서가 다른 부서로 이관 요청 및 처리 | 상 |
| - | 담당 부서가 답변 시 답변자 닉네임 + 부서명 함께 노출 | 상 |

### 4.6 포인트 · 뱃지
- 지식 공유 활동(질문, 답변, 채택, 첫 질문 등)에 포인트 부여
- 첫 질문, 첫 채택 답변, 답변 5개 이상 등 기본 뱃지 부여
- 포인트 누적 순위에 따라 랭킹 제공
- 첫 질문 등록 시 20P 등 튜토리얼성 보상 포함 (TUTORIAL_004)

### 4.7 알림 (NOTIFICATION)
- 채택, 답변 등록, 티켓 상태 변경 등 이벤트 발생 시 사용자에게 알림
- 알림 클릭 시 해당 게시글 또는 매뉴얼로 이동
- 모두 읽음 처리, 개별 알림 삭제 지원

### 4.8 FAQ / 메인
- FAQ_001: 인용수 많은 매뉴얼 + 좋아요 많은 워키를 메인에 요약
- FAQ_002: 인기 매뉴얼 (제목, 소제목, 개정일)
- FAQ_003: 인기 워키 (좋아요 기준)

### 4.9 ESG 시각화
- 사용자의 지식 공유 활동을 ESG 지표 카드로 시각화
- 고도화된 그래픽 시각화보다 측정 가능한 지표(지식 공유량, 예상 절감 시간, 출처 포함 답변 비율)를 우선한다.
- **ESG-S**: 구성원 간 지식 공유, 신입·비전문가 정보 접근성 향상
- **ESG-G**: 공식 규정 근거 제공, 답변 이력 관리, 관리자 검증, 투명한 지식 관리 체계

### 4.10 관리자 대시보드
- 메뉴: 대시보드 · 사용자 현황 · 오픈 질문 게시판 · 규정집(매뉴얼) 관리 · 티켓 관리
- 기본 데이터(사용자 수, 워키 질문 수, 티켓 수), 부서별 티켓 통계 확인
- 워키 게시글 삭제 권한은 **관리자 전용**
- 모든 관리자 작업은 admin_log 테이블에 기록

## 6. 사용자 시나리오 (요약 플로우)

1. **챗봇 → 워키 → 티켓** (만족 못 했을 때 단계적 에스컬레이션)
   - 사용자가 챗봇에 질문 → 로컬 임베딩 기반 검색 → 출처 포함 답변 → 답변 만족 못함/답변 없음 → 워키 질문 등록 안내 또는 티켓 발행 → 부서 답변/이관 → 채택 → 챗봇 검색 대상으로 재반영

2. **답변 → 채택 → 포인트 → 뱃지 → ESG 시각화**
   - 동료가 워키 질문에 답변 → 질문자가 채택 → 답변자 포인트 적립 → 누적 시 뱃지 부여 → ESG 시각화 진척

3. **관리자 운영**
   - 대시보드 진입 → 미답변 질문/장기 미처리 티켓 모니터링 → 매뉴얼 갱신, 부적절 게시글 삭제 → 모든 작업이 admin_log에 기록

## 7. 우선순위 및 단계

### Phase 1 (MVP, 우선순위 '상')
- 사용자 계정 (AUTH_001~004)
- 로컬 임베딩 기반 챗봇 RAG (KNOIT_001~005, 007, 008)
- 워키 게시판 (질문 등록·답변·반응·채택)
- 매뉴얼 등록·인용
- 티켓 발행, 부서 답변, 이관
- 알림 최소 구현
- 포인트/기본 뱃지
- ESG 지표 카드
- 관리자 대시보드 기본 기능

### Phase 2 (우선순위 '중/하')
- WebSocket/SSE 기반 완전 실시간 알림
- 포인트/뱃지 고도화
- FAQ / 인기 매뉴얼·워키 고도화
- ESG 그래픽 시각화
- 임베딩 배치(KNOIT_006)

### 미확정 정책 (확정 필요)
- 닉네임 중복 허용 범위 (전체 중복 불가 / 부서 내 중복 불가)
- 관리자 담당 범위 (인사팀 고정 / 부서별 관리자)
- 챗봇이 채택 답변 없는 경우 처리 (유사 질문 1~3개 참고 답변, 신뢰도 표시)
- 워키 답변 우선순위 알고리즘 (연관도 + 최신성 가중치)

## 8. 성공 지표 (KPI 제안)

| 지표 | 측정 방식 |
|---|---|
| 챗봇 답변 만족도 | 챗봇 답변 불만족 피드백 비율 |
| 워키 답변률 | 등록된 질문 중 채택 답변이 달린 비율 |
| 티켓 처리 SLA | 티켓 발행 → 처리완료 평균 소요 시간 |
| 매뉴얼 신뢰도 | 챗봇 답변 중 매뉴얼 인용 비율 |
| 참여도 | DAU/MAU, 월간 활성 질문자·답변자 수 |
| ESG 가치 | 누적 지식 공유 활동량(ESG-S 지표), 매뉴얼 갱신/검증 활동(ESG-G 지표) |

## 9. 범위 외 (Out of Scope)
- 외부(사외) 사용자 접근
- 실시간 음성 챗봇
- 외부 SaaS(예: Slack, MS Teams) 직접 통합 (추후 검토)
- 매뉴얼 외부 시스템 자동 동기화 (Phase 3 이후)
