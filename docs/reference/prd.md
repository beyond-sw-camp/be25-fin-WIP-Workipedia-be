## 1. 문서 개요

### 1.1 목적

Workipedia는 사내 구성원이 규정, 매뉴얼, 부서별 업무 지식, 기존 질의응답을 하나의 창구에서 검색하고 질문하며, 실제 처리가 필요한 업무 요청을 담당 부서의 티켓으로 연결하는 AI 기반 사내 지식 공유 플랫폼이다.

본 문서는 Workipedia 백엔드의 최종 제품 요구사항을 정의한다. 기존 초안의 요구사항을 현재 개발 환경과 구현 구조에 맞춰 정리하며, 기능 범위, 사용자 권한, 핵심 흐름, 비기능 요구사항, 우선순위, 성공 지표를 개발 기준으로 확정한다.

### 1.2 적용 범위

본 PRD는 다음 영역을 포함한다.

- 사용자 인증, 이메일 인증, JWT 기반 세션 관리
- 사용자, 팀 관리자, 시스템 관리자 권한 모델
- 매뉴얼, Worki Q&A, 지식화 데이터 기반 검색 및 RAG 질의응답
- AI 서버 연동, Qdrant 기반 벡터 검색, Cross-Encoder 재정렬, 출처 인용
- 업무 요청 티켓 생성, AI 부서 라우팅, 공통 접수함, 팀 배정, 처리 완료
- 처리 완료 티켓의 지식화 검토 및 승인
- 알림, 마이페이지, 포인트, 리더보드, ESG 지표
- Flash Chat 실시간 메시지
- 파일 첨부 및 Object Storage 연동
- 관리자 대시보드, AI Tool, AI Prompt, AI Sync 운영 기능

### 1.3 제품 한 줄 정의

Workipedia는 “질문은 지식을 찾고, 요청은 담당 부서로 연결하며, 처리 결과는 다시 조직 지식으로 축적하는” 사내 지식 순환 플랫폼이다.

## 2. 제품 배경과 문제 정의

### 2.1 문제

사내 지식은 매뉴얼 PDF, 부서 문서, 게시판, 메신저, 이메일, 개인 경험에 흩어져 있다. 구성원은 필요한 정보를 찾기 위해 여러 채널을 확인해야 하고, 담당자는 동일한 질문에 반복적으로 답변한다.

또한 실제 처리가 필요한 업무 요청은 메신저나 메일로 흘러가기 쉽다. 이 경우 요청 상태, 담당 부서, 처리 이력, 책임 소재가 명확하게 남지 않으며, 처리 결과가 조직 지식으로 재사용되지 않는다.

### 2.2 해결 방향

- 질문은 AI 챗봇 KnowIT, Worki Q&A, 매뉴얼 검색으로 빠르게 해결한다.
- AI 답변은 반드시 출처와 인용 정보를 함께 제공한다.
- 답변으로 해결되지 않는 요청은 티켓으로 전환해 담당 부서가 처리한다.
- AI는 개인 담당자를 직접 지정하지 않고 부서까지만 추천한다.
- 팀 관리자는 자기 부서 티켓을 확인하고 팀원에게 배정한다.
- 신뢰도 부족, AI 실패, 부서 불명확 요청은 공통 접수함으로 보낸다.
- 처리 완료된 요청은 팀 관리자 승인 후 지식화 데이터로 전환한다.
- 질문, 답변, 채택, 지식화, 처리 활동을 포인트와 ESG 지표로 시각화한다.

### 2.3 핵심 가치

- 탐색 시간 절감: 사내 지식을 자연어 질문으로 조회한다.
- 근거 기반 답변: 매뉴얼, Worki, 지식화 데이터 출처를 함께 제공한다.
- 요청 처리 추적: 실제 업무 요청을 티켓 상태로 관리한다.
- 조직 지식 축적: 처리 결과를 개인 경험이 아니라 재사용 가능한 지식으로 남긴다.
- 운영 투명성: 권한, 관리자 로그, 라우팅 로그, 인용 로그를 남긴다.

## 3. 개발 환경 및 기술 전제

### 3.1 백엔드 기술 스택

| 구분 | 기술 |
|---|---|
| Runtime | Java 21 |
| Framework | Spring Boot 3.5.14 |
| Build | Gradle |
| API | Spring MVC, WebFlux RestClient |
| Security | Spring Security, JWT |
| ORM | Spring Data JPA, Hibernate |
| Migration | Flyway |
| DB | MariaDB 11.4 |
| Cache / TTL | Redis 7.4 |
| Search | Elasticsearch 8.18.8 + Nori |
| Realtime | WebSocket/STOMP |
| Scheduler | Spring Scheduler, Quartz |
| Storage | S3 호환 Object Storage, R2, MinIO |
| Mail | Spring Mail |
| AI 연동 | 외부 AI 서버 HTTP API |
| Observability | Spring Actuator |

### 3.2 로컬 실행 환경

백엔드 로컬 기본 프로필은 `local`이다. 주요 인프라는 `docker-compose.yml`로 실행한다.

- MariaDB: 업무 데이터의 원본 저장소
- Redis: 캐시, TTL, 인증/임시 데이터, 실시간/휘발성 상태 보조
- Elasticsearch: Worki/매뉴얼 검색 인덱스
- AI 서버: `AI_BASE_URL`로 연동하며 기본값은 `http://localhost:8000`
- Object Storage: `STORAGE_PROVIDER`에 따라 R2, S3, MinIO 중 선택

### 3.3 데이터 원칙

- MariaDB를 업무 데이터의 원본으로 사용한다.
- Flyway 마이그레이션을 통해 스키마 변경을 관리한다.
- Elasticsearch와 Qdrant는 재구축 가능한 검색 인덱스로 취급한다.
- AI 서버의 Qdrant 벡터 저장소는 RAG 검색용이며, 원본 데이터 보존 책임은 BE DB에 둔다.
- 관리자 작업, 라우팅 판단, Tool 실행, 지식화 승인 등 운영상 중요한 행위는 로그로 남긴다.

## 4. 사용자와 권한

### 4.1 사용자 유형

| 사용자 | Role | 설명 |
|---|---|---|
| 일반 사용자 | `USER` | 질문, Worki 등록/답변, 티켓 발행, 본인 티켓 조회, 알림 확인 |
| 팀 관리자 | `TEAM_ADMIN` | 자기 부서 티켓 확인, 팀원 배정, 이관 요청, 지식화 승인 |
| 시스템 관리자 | `SYSTEM_ADMIN` | 공통 접수함 처리, 부서/R&R, 매뉴얼, AI Tool, Prompt, AI Sync, 운영 지표 관리 |

### 4.2 권한 원칙

- `USER`는 본인이 생성한 티켓과 본인 활동 정보를 조회한다.
- `TEAM_ADMIN`은 자기 부서에 배정된 티켓과 지식화 승인 대상을 관리한다.
- `TEAM_ADMIN`은 다른 부서로 직접 티켓을 넘기지 않고 이관 요청을 생성한다.
- `SYSTEM_ADMIN`은 공통 접수함, 부서 라우팅 기준, 전체 운영 설정을 관리한다.
- AI는 부서 추천까지만 수행하며 개인 담당자 배정은 팀 관리자가 수행한다.
- 전체 관리자 화면은 개인 평가가 아니라 서비스 운영 개선과 병목 파악을 목적으로 한다.

## 5. 핵심 사용자 시나리오

### 5.1 질문 해결

1. 사용자가 자연어 질문을 입력한다.
2. KnowIT이 매뉴얼, Worki, 지식화 데이터를 검색한다.
3. AI가 근거가 충분한 경우 출처와 함께 답변한다.
4. 사용자는 답변이 부족하면 Worki 질문 등록 또는 티켓 발행으로 전환한다.

### 5.2 Worki Q&A

1. 사용자가 공개 질문을 Worki에 등록한다.
2. 다른 구성원이 답변을 작성한다.
3. 질문자는 가장 적합한 답변을 채택한다.
4. 채택 답변은 이후 검색 및 RAG 후보로 활용된다.
5. 질문/답변/채택 활동에 포인트가 부여된다.

### 5.3 티켓 처리

1. 사용자가 실제 처리가 필요한 요청을 티켓으로 발행한다.
2. AI가 제목, 내용, 첨부, 부서 R&R, 기존 처리 이력을 근거로 후보 부서 Top 3를 산출한다.
3. 신뢰도와 점수 차이가 기준 이상이면 해당 부서로 자동 배정한다.
4. 기준 미달, AI 실패, 부서 불명확 요청은 공통 접수함으로 이동한다.
5. 팀 관리자는 부서 티켓을 팀원에게 배정한다.
6. 담당자는 처리 결과를 작성하고 완료 처리한다.
7. 팀 관리자는 지식화 필요 여부와 내용을 검토한다.

### 5.4 지식화

1. 처리 완료 티켓에서 개인정보와 민감정보를 제거한다.
2. AI가 재사용 가능한 지식 초안을 생성한다.
3. 팀 관리자가 초안을 검토하고 승인한다.
4. 승인된 데이터는 `knowledge_data` 계열 지식으로 저장되고 AI Sync를 통해 RAG 검색 대상에 반영된다.

### 5.5 Flash Chat

1. 사용자가 전사 공개 채널에 짧은 메시지를 작성한다.
2. 메시지는 WebSocket/STOMP로 실시간 전달된다.
3. 메시지는 정책 TTL에 따라 만료되며 영구 지식으로 저장하지 않는다.
4. 시스템 관리자는 TTL, 금칙어, 쿨다운 등 정책을 관리한다.

## 6. 기능 요구사항

본 장은 Workipedia 백엔드가 제공해야 하는 기능을 사용자 흐름과 구현 도메인 기준으로 세분화한다. 우선순위는 다음 기준을 따른다.

- `Must`: MVP와 핵심 시연에 반드시 필요한 기능
- `Should`: MVP 이후 고도화 또는 운영 편의에 필요한 기능
- `Could`: 확장 기능으로 분리 가능한 기능

### 6.1 인증, 계정, 보안 세션

| ID | 요구사항 | 우선순위 |
|---|---|---|
| AUTH-001 | 사용자는 사번, 이름, 부서, 회사 이메일, 비밀번호를 입력해 회원가입할 수 있다. | Must |
| AUTH-002 | 회원가입 시 회사 이메일 인증 코드를 발송하고, 인증 완료 후 계정을 생성한다. | Must |
| AUTH-003 | 이메일 인증 코드는 제한 시간과 재발송 정책을 가진다. | Must |
| AUTH-004 | 비밀번호는 영문, 숫자 조합과 길이 정책을 검증한다. | Must |
| AUTH-005 | 비밀번호는 BCrypt로 단방향 해시 저장한다. | Must |
| AUTH-006 | 사용자는 사번과 비밀번호로 로그인할 수 있다. | Must |
| AUTH-007 | 로그인 성공 시 Access Token과 Refresh Token을 발급한다. | Must |
| AUTH-008 | Access Token 만료 시 Refresh Token으로 토큰을 갱신할 수 있다. | Must |
| AUTH-009 | 로그아웃 시 서버에 저장된 Refresh Token을 무효화한다. | Must |
| AUTH-010 | 비활성, 탈퇴, 차단 상태의 사용자는 로그인할 수 없다. | Must |
| AUTH-011 | 사용자는 회사 이메일 인증을 통해 비밀번호 재설정 코드를 받을 수 있다. | Must |
| AUTH-012 | 사용자는 인증 코드 검증 후 새 비밀번호로 재설정할 수 있다. | Must |
| AUTH-013 | 인증 실패, 권한 부족, 토큰 만료 응답은 일관된 오류 형식으로 반환한다. | Must |

### 6.2 사용자, 부서, 권한

| ID | 요구사항 | 우선순위 |
|---|---|---|
| USER-001 | 사용자는 자신의 기본 프로필, 부서, 권한, 포인트 요약을 조회할 수 있다. | Must |
| USER-002 | 사용자는 마이페이지에서 본인이 작성한 질문, 답변, 티켓, 포인트 이력을 확인할 수 있다. | Must |
| USER-003 | 사용자 권한은 `USER`, `TEAM_ADMIN`, `SYSTEM_ADMIN`으로 구분한다. | Must |
| USER-004 | 부서는 이름, 설명, 활성 여부, 라우팅 기준 정보를 가진다. | Must |
| USER-005 | 팀 관리자는 하나 이상의 관리 대상 부서에 연결될 수 있다. | Should |
| USER-006 | 권한 검사는 role뿐 아니라 부서 소속과 티켓/문서 소유 관계를 함께 확인한다. | Must |
| USER-007 | 시스템 관리자는 사용자에게 팀 관리자 권한을 부여하거나 회수할 수 있다. | Should |

### 6.3 KnowIT 챗봇

| ID | 요구사항 | 우선순위 |
|---|---|---|
| CHAT-001 | 사용자는 자연어로 사내 규정, 업무 절차, 시스템 사용법을 질문할 수 있다. | Must |
| CHAT-002 | 챗봇은 사용자 질문과 최근 대화 맥락을 AI 서버에 전달한다. | Must |
| CHAT-003 | 챗봇 답변은 본문, 요약, 출처 목록, 후속 행동을 포함한다. | Must |
| CHAT-004 | 답변 근거가 부족하면 확정 답변 대신 Worki 질문 등록 또는 티켓 발행을 제안한다. | Must |
| CHAT-005 | AI 서버 장애, timeout, 차단 응답이 발생하면 사용자에게 실패 사유와 대체 경로를 제공한다. | Must |
| CHAT-006 | 사용자에게 반환되는 AI 답변에는 개인정보와 민감정보 마스킹 정책을 적용한다. | Must |
| CHAT-007 | 챗봇 답변에 사용된 출처는 RAG 인용 데이터로 저장한다. | Must |
| CHAT-008 | 답변 만족도 또는 부정 피드백을 수집해 운영 지표로 활용한다. | Should |

### 6.4 RAG 출처와 인용

| ID | 요구사항 | 우선순위 |
|---|---|---|
| RAG-001 | RAG 검색 대상은 매뉴얼, Worki, 지식화 데이터, 시스템 관리자 운영 지식으로 구분한다. | Must |
| RAG-002 | 검색 후보는 Vector Search 후 Cross-Encoder로 재정렬한다. | Must |
| RAG-003 | 답변 출처는 `sourceType`, `sourceId`, `chunkIndex`, 제목, 점수를 포함한다. | Must |
| RAG-004 | 출처 없는 답변은 공식 답변으로 제공하지 않는다. | Must |
| RAG-005 | 오래된 출처를 인용할 경우 최신성 경고를 표시할 수 있어야 한다. | Should |
| RAG-006 | RAG 인용 로그는 답변, 티켓 라우팅, 지식화 등 사용 위치를 구분해 저장한다. | Must |
| RAG-007 | 삭제 또는 비활성화된 원본은 신규 RAG 답변의 근거로 사용하지 않는다. | Must |
| RAG-008 | Qdrant 또는 AI 서버 오류 시 BE 원본 데이터의 무결성은 영향을 받지 않는다. | Must |

### 6.5 통합 검색

## 7. AI 서버 요구사항

### 7.1 책임 분리

BE는 사용자, 권한, 업무 데이터, 티켓 상태, 관리자 작업, 원본 저장소를 관리한다. AI 서버는 LLM, Embedding, Qdrant, Cross-Encoder, RAG 답변 생성, 티켓 부서 추천을 담당한다.

### 7.2 RAG 파이프라인

AI 서버는 다음 순서로 지식을 탐색한다.

1. 매뉴얼 RAG
2. Worki RAG
3. 지식화 데이터 RAG
4. 시스템 관리자 운영 지식
5. 허용된 Tool Calling
6. 실패 시 티켓 또는 Worki 전환 제안

각 단계는 `SUCCESS`, `NO_RESULT`, `ERROR`, `BLOCKED` 중 하나의 상태를 반환해야 한다.

### 7.3 티켓 라우팅

AI는 부서 후보 Top 3와 점수를 반환한다. 결정값은 다음 중 하나이다.

- `AUTO_ASSIGNED`: 기준 이상으로 특정 부서 자동 배정 가능
- `COMMON_QUEUE`: 신뢰도 부족, 후보 차이 부족, 실패, 검증 불가로 공통 접수함 이동

AI 점수는 모델별 상대값일 수 있으므로 `0~1` 정규화 값을 전제로 하지 않는다. BE는 점수와 점수 차이, decision, candidate departments, model, provider, reason을 라우팅 로그에 저장한다.

### 7.4 AI Sync

- BE는 변경된 원본 데이터에 대해 `ai_sync_jobs`를 생성한다.
- 동일 source의 중복 작업은 순서와 최종 상태를 보존한다.
- point ID는 source type, source id, chunk index 기반 deterministic ID를 사용한다.
- Qdrant collection은 데이터 유형과 생명주기에 따라 분리한다.
- Qdrant 장애 또는 데이터 유실 시 MariaDB 원본과 AI Sync Job으로 재구축 가능해야 한다.

## 11. 정책과 운영 기준

### 11.1 답변 정책

- AI는 근거가 부족하면 추측하지 않는다.
- 답변에는 가능한 한 매뉴얼, Worki, 지식화 데이터 출처를 포함한다.
- 오래된 출처를 사용하는 경우 최신성 경고를 표시한다.
- 개인정보가 포함될 가능성이 있으면 마스킹 또는 차단한다.

### 11.2 티켓 라우팅 정책

- AI는 후보 부서 추천과 근거 산출을 담당한다.
- 최종 개인 담당자 배정은 팀 관리자가 담당한다.
- 신뢰도 기준 미달, 점수 차이 부족, AI 장애는 공통 접수함으로 이동한다.
- 팀 관리자의 이관 요청은 시스템 관리자 검토 후 재배정한다.

### 11.3 지식 보존 정책

- Worki 질문과 채택 답변은 조직 지식으로 보존한다.
- 티켓 처리 내용은 승인 절차를 거친 경우에만 지식화한다.
- Flash Chat은 영구 지식이 아니며 TTL 이후 삭제한다.
- 파일 원본은 Object Storage에 두고 DB는 메타데이터와 권한만 관리한다.

## 13. 최종 제품 메시지

Workipedia는 단순한 AI 챗봇이 아니다. 사용자의 질문을 기존 지식으로 해결하고, 해결되지 않은 요청은 담당 부서가 처리하게 하며, 그 처리 결과를 다시 조직 지식으로 축적하는 사내 지식 운영 시스템이다.

질문은 기존 지식을 꺼내는 흐름이고, 요청은 새로운 지식을 만드는 흐름이다. Workipedia는 이 두 흐름을 하나의 제품 안에서 연결한다.
