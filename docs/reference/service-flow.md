## 1. 핵심 구조

Workipedia는 사용자의 입력을 **질문**, **요청**, **Flash Chat** 세 채널로 나눈다.

```text
질문 = 기존 지식을 찾고 답변을 남기는 기능
요청 = 담당 부서가 처리하고 새 지식 후보를 만드는 기능
Flash Chat = 10분 후 사라지는 가벼운 실시간 공개 질문 채널
```

```text
메인 진입
├─ 질문
│  └─ 워키/매뉴얼 기반 챗봇 검색
│     └─ 사내 지식/문서/FAQ 탐색
│
├─ 요청
│  └─ 티켓 발행
│     └─ 담당 부서 처리
│        └─ 답변 후 지식화 후보 등록
│           └─ 검수 후 워키 반영
│
└─ Flash Chat
   └─ 전사 공개 실시간 채팅
      └─ 10분 후 메시지 삭제
         ├─ 해결되면 종료
         ├─ 기록할 가치가 있으면 워키 질문으로 전환
         └─ 공식 처리가 필요하면 요청 티켓으로 전환
```

질문 영역의 목적은 담당자 처리가 아니라 정보 탐색이다. 단순 정보성 질문은 티켓으로 보내지 않고, 매뉴얼·워키·챗봇으로 해결하는 것이 원칙이다.

요청 영역의 목적은 실제 업무 처리다. 요청 티켓은 기존 메일 기반 업무 요청을 대체하는 공식 처리 단위이며, 처리 결과는 지식화 후보가 될 수 있다.

Flash Chat 영역의 목적은 가장 빠른 동료 간 확인이다. 메시지는 Redis TTL로 자동 삭제되며, 영구 지식이나 공식 처리 이력으로 보존하지 않는다. 다만 반복될 만한 내용은 워키 질문으로, 담당 부서 처리가 필요한 내용은 요청 티켓으로 전환할 수 있다.

## 2. 세 채널의 차이

| 구분 | 질문 | 요청 | Flash Chat |
|---|---|---|---|
| 목적 | 정보 탐색, 답변 축적 | 공식 업무 처리 | 빠른 동료 확인 |
| 무게감 | 가벼움~중간 | 공식 요청 | 가장 가벼움 |
| 기록 | 영구 보존 | 영구 보존 | 10분 후 삭제 |
| 사용자 표시 | 닉네임 기반 가능 | 부서 기반이 자연스러움 | 닉네임 기반 |
| 처리 주체 | 챗봇/워키/매뉴얼 | 담당 부서 | 전사 구성원 |
| 결과 | 답변 제공, 워키 축적 | 티켓 처리 | 즉시 답변 또는 전환 |
| 지식화 | 기존 워키 검색/답변 축적 | 처리 답변이 새 지식 후보가 됨 | 직접 지식화하지 않음. 필요 시 워키/티켓으로 전환 |

## 3. 역할 정의

| Role | 조회 범위 | 주요 역할 |
|---|---|---|
| `USER` | 본인 질문, 본인 요청 티켓, Flash Chat 활성 메시지 | 질문, 요청, Flash Chat 메시지/답장 작성, 상태 확인 |
| `TEAM_ADMIN` | 자기 팀 티켓 | 티켓 확인, 팀원 배정, 답변 관리, 지식화 승인 |
| `SYSTEM_ADMIN` | 공통 접수 큐, 운영 메타데이터 | 자동 배정 실패 검토, 부서/카테고리 매핑, 전체 운영 현황 확인 |

`SYSTEM_ADMIN`은 모든 티켓 본문을 무제한으로 보는 역할이 아니다. 대시보드는 개인 평가가 아니라 서비스 운영 개선을 목적으로 하며, 팀 단위 처리 현황·공통 접수 큐·자동 배정 성공률·지식화 전환 건수 같은 운영 지표를 중심으로 제공한다.

## 4. 티켓 라우팅 규칙

요청 티켓은 AI가 담당 부서를 우선 예측한다.

```text
사용자 요청
→ AI가 담당 부서 예측
→ 신뢰도 높음
→ 해당 부서 큐로 자동 배정
```

신뢰도가 낮으면 특정 부서에 바로 보내지 않고 공통 접수 큐로 보낸다.

```text
사용자 요청
→ AI가 담당 부서 예측
→ 신뢰도 낮음
→ 공통 접수 큐
→ SYSTEM_ADMIN 또는 운영 관리자가 수동 배정
```

신뢰도는 다음 요소를 조합해 계산한다. 아래 가중치와 임계값은 MVP 초기 기준이며, 운영 데이터가 쌓이면 조정할 수 있다.

| 기준 | 가중치 |
|---|---:|
| 요청 키워드 | 25 |
| 관련 워키/매뉴얼 유사도 | 30 |
| 카테고리-부서 매핑 | 20 |
| 과거 유사 티켓 처리 부서 | 15 |
| LLM 분류 결과 | 10 |

```text
routing_confidence_score =
  keyword_score * 0.25
+ document_similarity_score * 0.30
+ category_mapping_score * 0.20
+ past_ticket_score * 0.15
+ llm_classification_score * 0.10
```

| 신뢰도 | 처리 방식 |
|---:|---|
| 80 이상 | 자동으로 담당 부서 배정 |
| 50 이상 80 미만 | 후보 부서 추천 후 관리자 검토 |
| 50 미만 | 공통 접수 큐 또는 추가 정보 요청 |

## 5. 지식화 규칙

티켓을 처리한 담당자는 처리 완료 시 지식화 후보 등록 여부를 선택한다.

```text
티켓 처리 완료
→ 담당자가 지식화 후보 등록 여부 선택
→ AI가 답변을 워키 초안으로 요약
→ TEAM_ADMIN이 검수
→ 공식 워키 등록
```

티켓 원문을 그대로 워키로 올리면 안 된다. 개인 사례와 민감 정보를 제거하고 일반 절차 형태로 바꿔야 한다.

| 원본 티켓 | 지식화 문서 |
|---|---|
| 제 노트북 자산번호 H1234를 금요일에 반납할 수 있나요? | 노트북 반납은 자산관리팀을 통해 진행하며, 반납 전 자산번호 확인과 반납 신청이 필요합니다. |

---

## 6. 티켓 이관 규칙

이관은 담당 팀의 `TEAM_ADMIN`만 요청할 수 있다. 팀원이 임의로 이관하지 않는다.

이관 요청 시 티켓은 다른 부서로 직접 이동하지 않고 **공통 접수 큐**로 돌아간다. 이후 `SYSTEM_ADMIN`이 이관 사유와 후보 부서를 확인한 뒤 적절한 담당 부서로 다시 배정한다.

```text
TEAM_ADMIN이 이관 요청
→ 티켓 상태 COMMON_QUEUE
→ SYSTEM_ADMIN이 이관 사유 확인
→ 담당 부서 재배정
→ 새 부서 TEAM_ADMIN이 팀 큐에서 확인
→ 팀원 배정
```

## 7. Flash Chat 규칙

Flash Chat은 전사 단일 공개 채널이다. 입장 절차는 별도로 두지 않고, 인증된 사용자는 현재 활성 메시지를 조회한 뒤 STOMP topic을 구독한다.

```text
사용자 Flash Chat 진입
→ 현재 활성 메시지 조회
→ /topic/flash-chat 구독
→ 메시지/답장 송수신
→ Redis TTL 만료 시 메시지 삭제
```

| 항목 | 기준 |
|---|---|
| 저장소 | Redis |
| 기본 TTL | 600초 |
| 영구 DB 저장 | 하지 않음 |
| AI 개입 | 기본 없음 |
| 답장 | 특정 메시지 기준 reply 가능 |
| 반응 | 좋아요는 MVP 이후 |
| 관리자 설정 | TTL, 전송 쿨다운, 금지어, 강제 삭제 |
| 관리자 기록 | 정책 변경과 강제 삭제를 `admin_logs`에 기록 |

Flash Chat에서 나온 내용이 반복 질문으로 남길 가치가 있으면 워키 질문으로 옮긴다. 담당 부서의 처리나 공식 답변이 필요하면 요청 티켓으로 전환한다.

## 8. 대표 시나리오

### 시나리오 A. 질문 → 챗봇 → 답변(매뉴얼 + 워키)

```text
사용자가 "연차 신청은 어디서 하나요?" 질문
→ 챗봇이 매뉴얼과 워키 검색
→ 매뉴얼 답변: HR 시스템 신청 경로와 승인 절차
→ 워키 답변: 실무자가 남긴 신청 팁
→ 출처 링크 제공
→ 사용자가 해결
```

### 시나리오 B. 질문 → 챗봇 → 티켓 발행

```text
사용자가 "해외 출장 중 의료비 처리는 어떻게 하나요?" 질문
→ 챗봇이 관련 문서를 찾지 못함
→ 사용자가 "담당 부서에 요청하기" 선택
→ 질문 내용이 요청 초안으로 전환
→ 티켓 발행
→ AI가 담당 부서와 신뢰도 산출
```

### 시나리오 C. 요청 → 자동 배정 → 부서장 확인 → 팀원 분배 → 처리 완료

```text
사용자가 "VPN 접속 오류 처리 요청합니다." 요청 작성
→ AI가 IT지원팀으로 자동 배정(신뢰도 87)
→ IT지원팀 TEAM_ADMIN이 팀 큐에서 티켓 확인
→ 담당 팀원에게 티켓 배정
→ 팀원이 조치 결과 작성
→ 티켓 처리 완료
→ 사용자에게 완료 알림
```

### 시나리오 D. 요청 → 자동 배정 → 처리 완료 → 지식화

```text
사용자가 "노트북 반납 신청합니다." 요청 작성
→ AI가 자산관리팀으로 자동 배정(신뢰도 91)
→ 자산관리팀 TEAM_ADMIN이 담당 팀원에게 분배
→ 팀원이 반납 절차 안내 후 처리 완료
→ 담당자가 지식화 후보 등록 선택
→ AI가 개인 정보를 제거하고 일반 절차 문서 초안 생성
→ TEAM_ADMIN이 검수
→ 워키에 "노트북 반납 절차" 문서 반영
→ 이후 같은 질문은 챗봇이 워키 근거로 답변
```

### 시나리오 E. 요청 → 신뢰도 낮음 → 공통 접수 큐 → 전체 담당자 분배 → 부서 처리

```text
사용자가 "SSD 추가 장착 가능 여부와 보안씰 처리를 확인받고 싶습니다." 요청 작성
→ AI가 자산관리팀/정보보안팀 후보를 찾지만 신뢰도 63
→ 특정 부서 자동 배정 대신 공통 접수 큐로 이동
→ SYSTEM_ADMIN이 내용과 후보 근거 확인
→ 담당 부서를 자산관리팀으로 수동 배정하고 보안팀 협조 메모 추가
→ 자산관리팀 TEAM_ADMIN이 팀 큐에서 티켓 확인
→ 담당 팀원에게 티켓 분배
→ 팀원이 요청 처리 완료
```

### 시나리오 F. 요청 → 부서 확인 → 이관 요청 → 공통 접수 큐 → 재배정

```text
사용자가 "계약서 검토 요청합니다." 요청 작성
→ AI가 경영지원팀으로 자동 배정
→ 경영지원팀 TEAM_ADMIN이 팀 큐에서 확인
→ 본인 부서 처리가 아니라고 판단
→ TEAM_ADMIN이 이관 사유와 함께 이관 요청
→ 티켓이 공통 접수 큐로 이동
→ SYSTEM_ADMIN이 이관 사유를 확인하고 법무팀으로 재배정
→ 법무팀 TEAM_ADMIN이 팀 큐에서 확인
→ 담당 팀원에게 티켓 분배
→ 팀원이 요청 처리 완료
```

### 시나리오 G. Flash Chat → 즉시 해결

```text
사용자가 Flash Chat에 "회의실 A 빔프로젝터 리모컨 어디 있나요?" 작성
→ 동료가 "A 회의실 앞 수납함 두 번째 칸에 있어요" 답변
→ 사용자가 해결
→ 메시지는 TTL 만료 후 삭제
```

### 시나리오 H. Flash Chat → 워키 질문 전환

```text
사용자가 Flash Chat에 "반차랑 연차 차이가 정확히 뭐예요?" 작성
→ 여러 사용자가 답변하지만 반복 질문으로 남길 가치가 있음
→ 사용자가 워키 질문으로 등록
→ 답변 채택 후 이후 챗봇/워키 근거로 재사용
```

### 시나리오 I. Flash Chat → 요청 티켓 전환

```text
사용자가 Flash Chat에 "노트북 화면이 안 켜지는데 바로 처리 가능할까요?" 작성
→ 동료 답변만으로 해결되지 않음
→ 사용자가 요청 티켓으로 전환
→ AI가 담당 부서와 신뢰도 산출
→ 담당 부서 큐 또는 공통 접수 큐로 이동
```

## 9. ESG 지표 연결

질문, 요청, Flash Chat 흐름은 다음 지표로 측정한다.

| 지표 | 산출 의도 |
|---|---|
| 챗봇 해결 건수 | 반복 문의 감소 |
| Flash Chat 해결 건수 | 빠른 동료 확인으로 공식 요청을 줄인 정도 |
| Flash Chat → 워키/티켓 전환 건수 | 임시 대화가 영구 지식 또는 공식 처리로 전환된 정도 |
| 예상 절감 시간 | 기존 문의 처리 시간 대비 챗봇 응답 시간 절감 |
| 지식화 전환 건수 | 처리 결과가 조직 지식으로 축적된 양 |
| 지식화 문서 재사용 횟수 | 한 번 처리한 요청이 이후 질문 해결에 재사용된 정도 |
| 자동 배정 성공률 | 요청 라우팅 효율 |
| 공통 접수 큐 비율 | AI가 확신하지 못한 요청 비율 |

```text
누적 절감 시간 =
(기존 평균 문의 처리 시간 - 챗봇 평균 응답 시간) × 챗봇 해결 건수
```

```text
지식화 절감 시간 =
지식화 문서 조회/활용 횟수 × 기존 평균 반복 문의 처리 시간
```
