## 1. 발표 방향

중간 발표에서는 Workipedia를 단순 AI 챗봇으로 설명하면 약하다.

핵심 포지셔닝은 아래 문장으로 고정한다.

> Workipedia는 단순 AI 챗봇이 아니라, 사내 질문을 AI가 1차 응답하고, 해결되지 않은 질문은 워키와 티켓으로 연결하며, 검증된 답변을 다시 조직 지식으로 축적하는 B2B 사내 지식 운영 플랫폼이다.

발표의 목표는 “AI가 답변한다”가 아니라 “사내 문의 업무가 어떻게 빨라지고, 정확해지고, 조직 지식으로 남는가”를 보여주는 것이다.

## 2. 받은 피드백과 보완 방향

| 피드백 | 보완 방향 | 발표에 넣을 자료 |
|---|---|---|
| 문제 정의가 약함 | 사용자별 문제와 손실을 명확히 정의 | 사용자/문제/손실 표 |
| 서비스 정체성이 모호함 | 챗봇이 아니라 지식 운영 플랫폼으로 정의 | 한 문장 서비스 정의 |
| 단순 챗봇처럼 보임 | 워키, 티켓, 관리자 검수, FAQ 자동화를 연결 | 업무 흐름 다이어그램 |
| 빠른 처리와 정확한 처리 구분 필요 | 문의 유형별 처리 정책 제시 | 처리 분기표 |
| ESG 수치 근거 필요 | 절감 시간/출력량/운영비 산식 제시 | ESG 계산식 |
| 차별성 부족 | B2B 업무 특화 비교표 제시 | 일반 챗봇 vs Workipedia |

## 3. 문제 정의

### 3.1 핵심 문제

사내 정보는 존재하지만, 직원이 필요한 순간에 신뢰 가능한 답을 빠르게 찾기 어렵다.

문제는 단순 검색 실패가 아니라 아래 세 가지가 동시에 발생한다는 점이다.

1. 정보가 여러 문서, 메신저, 게시판, 담당자에게 흩어져 있다.
2. 같은 질문이 반복되어 담당 부서의 응답 비용이 증가한다.
3. 한 번 해결된 답변이 조직 지식으로 축적되지 않는다.

### 3.2 사용자별 문제

| 사용자 | 겪는 문제 | 현재 방식 | 발생 손실 |
|---|---|---|---|
| 신입사원 | 규정, 복지, 신청 절차를 모름 | 메신저/동료에게 질문 | 온보딩 지연, 반복 질문 |
| 일반 직원 | 담당 부서와 공식 절차를 모름 | 사람에게 물어보거나 문서 검색 | 업무 지연, 잘못된 정보 사용 |
| 담당 부서 | 같은 질문에 반복 답변 | 메일/메신저/구두 응답 | 반복 업무 증가 |
| 관리자 | 문의/답변 이력 관리 어려움 | 흩어진 문서와 채팅 확인 | 지식 축적 실패, 감사 어려움 |

### 3.3 데이터 기반 근거 확보 방법

팀 내부 또는 주변 사용자 5~10명을 대상으로 간단한 설문을 진행한다.

추천 질문:

| 질문 | 측정 목적 |
|---|---|
| 사내 규정/절차를 찾는 데 평균 몇 분 걸리나요? | 탐색 시간 |
| 담당 부서를 몰라 문의가 지연된 경험이 있나요? | 업무 지연 |
| 같은 질문을 반복해서 받은 경험이 있나요? | 반복 문의 |
| 메신저/메일에 남은 답변을 다시 찾기 어려웠던 적이 있나요? | 지식 재사용 실패 |
| AI 답변에 출처가 없으면 신뢰할 수 있나요? | 출처 필요성 |

발표에서는 실제 설문이 부족하면 “팀 내부 예비 설문” 또는 “운영 시 측정할 지표”로 표현한다.

## 4. 서비스 정의

### 4.1 한 문장 정의

> Workipedia는 사용자의 입력을 질문과 요청으로 나누어, 질문은 매뉴얼·워키 기반 챗봇으로 빠르게 해결하고 요청은 담당 부서 티켓으로 처리한 뒤 지식화하는 사내 지식 운영 플랫폼이다.

### 4.2 서비스 범위

| 포함 | 설명 |
|---|---|
| AI 챗봇 | 매뉴얼/워키 기반 출처 포함 답변 |
| 워키 게시판 | 기존 지식과 동료 답변을 축적하는 Q&A 공간 |
| 요청 티켓 | 공식 판단이나 담당 부서 처리가 필요한 요청 연결 |
| 공통 접수 큐 | AI 배정 신뢰도가 낮은 요청을 운영 관리자가 분류 |
| 지식화 | 처리 완료 요청을 개인 정보 제거 후 워키 후보로 전환 |
| FAQ | 자주 묻는 질문과 인기 문서 자동 노출 |
| 알림 | 답변, 채택, 티켓 상태 변경 알림 |
| 포인트 | 지식 공유 참여 보상 |
| 관리자 대시보드 | 사용자, 질문, 티켓, 포인트, 운영 현황 관리 |

| 제외 또는 후순위 | 이유 |
|---|---|
| ERP/HR 직접 연동 | MVP 범위 초과 |
| 실시간 음성 챗봇 | 발표 목표와 무관 |
| 고도화된 ESG 시각화 | 수치 근거 확보 후 진행 |
| 완전 자동 승인 처리 | 업무 리스크가 높아 관리자/담당 부서 검토 필요 |

## 5. 핵심 업무 흐름

발표에서는 아래 흐름을 반드시 보여준다.

```text
질문
-> 챗봇이 매뉴얼/워키 기반 검색
-> 출처 포함 답변 제공
-> 해결

요청
-> 티켓 발행
-> AI가 라우팅 신뢰도 계산
-> 신뢰도 높음: 담당 부서 큐 자동 배정
-> 신뢰도 낮음: 공통 접수 큐에서 운영 관리자 검토
-> 부서장이 팀원에게 티켓 분배
-> 부서가 맞지 않으면 TEAM_ADMIN이 이관 요청
-> 이관 요청된 티켓은 공통 접수 큐에서 재배정
-> 담당자가 요청 처리 완료
-> 필요 시 개인 내용을 일반화해 지식화 후보 등록
-> 팀 관리자 검수 후 워키 반영
```

## 6. 빠른 처리와 정확한 처리 구분

심사위원이 중요하게 본 부분이므로, 문의 유형별 처리 정책을 명확히 보여준다.

| 문의 유형 | 처리 방식 | 예시 | 이유 |
|---|---|---|---|
| 단순 정보 문의 | AI 즉시 답변 | 휴가 신청 위치, 복지몰 링크 | 빠른 처리 우선 |
| 반복 FAQ | AI 답변 + FAQ 추천 | 출장비 기준, 교육 신청 | 반복 문의 감소 |
| 개인정보 포함 | 마스킹 또는 답변 거부 | 연락처, 계좌, 주민번호 | 개인정보 보호 |
| 실제 처리 요청 | 티켓 전환 | 권한 신청, 장비 반납, VPN 오류 | 담당 부서 처리 필요 |
| 공식 판단 필요 | 티켓 전환 | 법인카드 예외 승인, 법무 검토 | 정확성/책임 우선 |
| 규정 변경 가능성 있음 | 관리자 검수 필요 | 인사 규정, 징계, 계약 | 최신성/공식성 필요 |
| 담당 부서 불명확 | 공통 접수 큐 | SSD 장착과 보안씰 동시 문의 | 무리한 자동 배정 방지 |

핵심 메시지:

> 모든 문의를 AI가 자동 처리하지 않는다. 빠르게 처리해도 되는 문의는 AI가 처리하고, 책임과 정확성이 필요한 문의는 티켓과 관리자 검수로 넘긴다.

## 7. 단순 챗봇 이상의 가치

| 기능 | 단순 챗봇 대비 가치 |
|---|---|
| 출처 포함 답변 | 답변 신뢰성 확보 |
| 워키 전환 | 답변 실패를 막다른 길로 두지 않음 |
| 티켓 전환 | 공식 판단이 필요한 업무를 담당 부서로 연결 |
| 자동 배정/공통 접수 큐 | 확실한 요청은 빠르게 보내고 애매한 요청은 사람이 검토 |
| TEAM_ADMIN 이관 요청 | 잘못 배정된 티켓을 공통 접수 큐로 되돌려 재분류 |
| 지식화 | 처리 완료 답변을 일반 절차로 바꿔 다음 질문 해결에 재사용 |
| 관리자 검수 | 부적절한 지식과 오래된 정보를 관리 |
| FAQ 자동화 | 반복 문의를 메인/FAQ로 재노출 |
| 포인트 | 지식 공유 참여 유도 |
| 이력 보존 | 답변/채택/티켓 처리 이력을 조직 자산으로 보존 |

## 8. 차별성과 경쟁력

| 구분 | 일반 AI 챗봇 | Workipedia |
|---|---|---|
| 목적 | 질문에 답변 | 사내 문의 처리와 지식 축적 |
| 답변 방식 | 생성형 답변 중심 | 출처 기반 RAG |
| 실패 처리 | 답변 실패 또는 재질문 | 질문/요청 전환, 워키/티켓 연결 |
| 업무 연결 | 약함 | 담당 부서 티켓 큐와 팀원 배정 |
| 지식 축적 | 제한적 | 처리 완료 요청의 지식화 |
| 권한 관리 | 단순 | USER/TEAM_ADMIN/SYSTEM_ADMIN |
| 관리자 기능 | 제한적 | 팀 큐, 공통 접수 큐, 검수, admin log |
| B2B 특화 | 약함 | 부서, 공식 답변, 감사, 이력 보존 |

발표 문장:

> Workipedia의 차별점은 답변 생성이 아니라, 답변 이후의 업무 흐름과 지식 재사용 구조까지 설계했다는 점이다.

## 9. ESG와 효율 지표

ESG는 감성적 표현보다 측정 가능한 지표로 제시한다.

### 9.1 운영 효율 계산식

```text
월 반복 문의 감소 시간 =
월 반복 문의 건수 x 평균 응답 시간
```

예시:

```text
월 300건 반복 문의 x 건당 5분 = 월 1,500분
월 1,500분 = 월 25시간 절감
```

### 9.2 비용 절감 계산식

```text
월 운영 비용 절감 =
월 반복 문의 감소 시간 x 담당자 평균 시간당 비용
```

예시:

```text
월 25시간 x 시간당 20,000원 = 월 500,000원 절감
```

### 9.3 문서 출력 절감 계산식

```text
월 출력 절감 페이지 =
월 디지털 조회 건수 x 평균 출력 페이지 수 x 출력 대체율
```

예시:

```text
월 500회 조회 x 평균 2페이지 x 출력 대체율 30% = 월 300페이지 절감
```

### 9.4 ESG-S/G 지표

| ESG | 지표 | 측정 방식 |
|---|---|---|
| S | 지식 공유량 | 질문 수, 답변 수, 채택 수 |
| S | 신입/비전문가 접근성 | 챗봇 질문 해결률, 워키 답변률 |
| G | 검증 가능한 답변 | 출처 포함 답변 비율 |
| G | 투명한 운영 | admin_logs 기록 수, 티켓 처리 이력 |
| G | 문서 최신성 | 매뉴얼 갱신 수, FAQ 반영 수 |

주의:

- 발표에서 ESG를 크게 주장하려면 수치 근거가 필요하다.
- 수치가 부족하면 “운영 후 측정 가능한 ESG 지표 설계”로 표현한다.

## 10. 발표 구성안

### 10.1 권장 흐름

| 순서 | 내용 | 핵심 메시지 |
|---|---|---|
| 1 | 문제 정의 | 사내 지식은 있지만 찾기 어렵고 반복 문의가 발생한다 |
| 2 | 사용자별 문제 | 신입, 일반 직원, 담당 부서, 관리자 모두 다른 문제를 겪는다 |
| 3 | 서비스 정의 | 단순 챗봇이 아니라 사내 지식 운영 플랫폼이다 |
| 4 | 핵심 업무 흐름 | 질문 -> 챗봇 답변, 요청 -> 티켓 처리 -> 지식화 |
| 5 | 처리 정책 | 빠른 처리와 정확한 처리를 분리한다 |
| 6 | 차별성 | B2B 업무 특화, 출처, 권한, 관리자 검수 |
| 7 | ESG/효율 지표 | 반복 문의 시간과 운영 비용을 측정한다 |
| 8 | 구현 범위/WBS | 6월 26일 배포, 7월 3일 발표 |

### 10.2 시연 흐름

```text
로그인
-> 질문 입력
-> 챗봇의 매뉴얼 + 워키 출처 포함 답변 확인
-> 공식 처리 필요 질문을 요청 티켓으로 전환
-> 요청 티켓 자동 배정 확인
-> 부서장이 팀원에게 티켓 분배
-> 부서가 맞지 않은 티켓은 TEAM_ADMIN이 이관 요청
-> 공통 접수 큐에서 전체 관리자가 재배정
-> 담당자가 요청 처리 완료
-> 처리 완료 답변을 지식화 후보로 등록
-> 팀 관리자 검수 후 워키 반영
-> 라우팅 신뢰도 낮은 요청은 공통 접수 큐에서 전체 관리자가 수동 배정
-> ESG/운영 대시보드에서 절감 시간, 지식화 전환율, 자동 배정 성공률 확인
```

## 11. 발표에서 피해야 할 표현

| 피할 표현 | 이유 | 대체 표현 |
|---|---|---|
| AI가 다 답변합니다 | 신뢰성/책임 문제 | AI가 1차 응답하고, 공식 판단은 티켓으로 연결합니다 |
| ESG 효과가 있습니다 | 근거 부족 | 운영 데이터 기반으로 ESG-S/G 지표를 측정합니다 |
| 사내 모든 시스템과 연동합니다 | MVP 과대 범위 | MVP는 지식 문의 흐름에 집중하고, 외부 연동은 후속 범위입니다 |
| 챗봇 서비스입니다 | 범위 축소 | 사내 지식 운영 플랫폼입니다 |
| 자동 승인합니다 | 업무 리스크 | 관리자/담당 부서 검수 흐름을 둡니다 |
